# Runoff v1.5 — Golden Ownership: Unify + Bind (Design)

Date: 2026-07-19 · Status: approved for planning · Predecessors: v1.4 parse plans (`2026-07-19-runoff-v1.4-parse-plans-design.md`), v1.3b run pipeline, v1.2a goldens.

## 1. Goal

Own golden examples the way v1.4 owns source data. Two LLM rounds give every golden a structured, data-aware representation:

1. **Unify** (upload-time): any exemplar file becomes a universal `RunDocument` — the same shape generated reports use — with a detected period. No bindings yet.
2. **Bind to data** (auto after unify, re-triggerable): an agent inventories every narration-driving value and every table in the document and binds each to warehouse data (`familyId` + `:period`-parameterized SQL). The app re-executes every binding to verify it. Per-golden **boundness** (bound/total) is displayed; only exemplar goldens can be partially bound — run/section goldens derive their inventory deterministically from their citations.

The bound representation becomes **the** abstraction layer: blueprint-building (copilot) and report-generation agents consume goldens only through it. Primary purpose: **blueprint scaffolding** — copilot lifts inventory SQL into section queries (`update_section_queries` already exists).

Design principles carried over from v1.4: LLM artifacts are validated zod documents, never trusted free text; verification is executed by the app, not claimed by the agent; steady state spends zero or few LLM tokens by reusing prior artifacts (here: sibling goldens' inventories); byte-exact error strings; plan-less/golden-less behavior unchanged.

## 2. Data model

`goldens` gains four nullable TEXT columns (after `note`): `period`, `document`, `unify_error`, `bindings`. Breaking schema, no migration — upgrade is `rm -f data/runoff.db && rm -rf data/warehouses && pnpm seed`. Drizzle mirror updated in step.

New zod schemas in `packages/core/src/types/document.ts` (alongside the existing interfaces, which they must match field-for-field):

- `CitationSchema`, `SpanSchema`, `BlockSchema`, `DocSectionSchema`, `RunDocumentSchema` — validation for the existing `RunDocument` shape. Section keys must match `^[a-z][a-z0-9_]*$` and be unique within a document.
- `BindingItemSchema`:
  ```
  {
    id: string (^[a-z][a-z0-9_]*$, unique within inventory),
    kind: "value" | "table",
    anchor: { sectionKey: string, blockIndex: int ≥ 0, spanIndex: int ≥ 0 | null },
    raw: string (1–200 chars),            // the text as it appears: "$4.2M", or a table caption/header summary
    parsed: number | string | null,       // normalized: 4200000, "EMEA"
    binding: {
      familyId: string,
      sql: string,                        // may reference :period
      verifiedValue: number | string | null,
      status: "bound" | "mismatch" | "error"
    } | null,
    reason: string | null                 // required iff binding is null or status ≠ "bound"
  }
  ```
- `BindingInventorySchema`: `{ version: 1, items: BindingItem[] (max 60) }`.
- `SubmittedItemSchema` / `SubmittedInventorySchema`: the agent-facing form — identical, except `binding` is `{ familyId, sql } | null` (no `verifiedValue`/`status`; the app stamps those in verification). `submit_inventory` validates against this form; `verifyInventory` (§6) turns a submitted inventory into a stored `BindingInventory`.
- `validateInventoryAnchors(inventory, document)` throws on the first bad anchor: unknown sectionKey, blockIndex out of range, spanIndex present on a table block or out of range, `kind: "table"` anchoring a non-table block (and vice versa). Error strings in §11.

Boundness is always derived — `items.filter(i => i.binding?.status === "bound").length / items.length` — never stored. An empty inventory (0 items) renders as “nothing to bind”, not 100%.

## 3. Unify round (exemplars only, upload-time)

Engine `packages/engine/src/unifyGolden.ts`:

- `unifyGoldenReport({ client, filename, text }): Promise<{ document: RunDocument, period: string | null } | null>` — single structured call following the classify/propose pattern: `response_format: {type: "json_object"}`, max 4000 completion tokens, 2 attempts, zod-validated (`RunDocumentSchema` + section-key rules), null on final failure. The injected client is the only I/O; no env reads.
- Prompt contract (`UNIFY_CONTRACT`): restructure the report into `RunDocument` JSON — sections with headings, paragraphs as spans (**no citations** — spans carry text only at this stage), tabular content as table blocks. **Preserve the original wording verbatim wherever possible; restructure, never paraphrase.** Detect the period the report covers (`period` field, or null): must match one of the canonical forms (`2026-Q1` / `2026-06` / `2026`) or be null — enforced by validation, not trust.
- Degeneracy self-check (one automatic retry, mirroring v1.4): degenerate iff `sections.length === 0` or every block is empty (no spans / no rows). The retry appends the failure reason to the prompt.
- Input text comes from the existing sourcePack extractors (PDF/docx/text/markdown), capped at 24,000 chars with head+tail sampling (first 20k + last 4k) when longer. CSV/XLSX exemplars are not unified (reports, not data): unify records `unify_error` = `unsupported exemplar type for unify: <mime>`.

The exemplar upload route runs extract → unify → store `document`+`period` (or `unify_error`) → **auto-chains the bind round** (§5). Unify failure keeps the golden with its raw file and a retry affordance; a failed-unify golden is **inert to agents** — there is no flat-text fallback (see §8).

## 4. Deterministic inventory for run/section goldens

Run and section goldens are born structured: `document` materializes from the run row at read time (never copied into `goldens`). Their inventory needs no LLM:

- `inventoryFromCitations(document, catalog): BindingInventory` (engine) walks the doc; every span with a citation becomes a `value` item (`raw` = span text, `parsed` = numeric parse of the span where possible, binding SQL = `compileLocator(citation.locator)` output, familyId = citation.sourceId). Table blocks become `table` items bound via the section's baked queries when a query covers that table's family, else unbound with reason `no query covers this table`.
- Verification (§6) then executes and stamps statuses exactly as for exemplars.
- Starring a run/section golden computes and verifies this inventory immediately; the UI button for these kinds is “verify bindings” (deterministic re-check, zero LLM).

## 5. Bind round (exemplars: agentic, auto after unify, re-triggerable with feedback)

Engine `packages/engine/src/bindGolden.ts`, shaped like `copilotTurn`’s tool loop:

- `bindGolden({ client, catalog, runSql, document, period, siblings, feedback?, priorInventory? }): Promise<SubmittedInventory | null>` — returns the agent's submitted inventory; the caller runs `verifyInventory` (§6) before storing.
- System prompt: task contract + serialized data catalog (existing `serializeCatalog`) + the golden’s period (or “unknown”) + the unified document (compact rendering) + **sibling context** (§5a) + prior inventory and feedback when re-running.
- Tools:
  - `run_sql` — the existing read-only executor contract (200-row/10k-char caps, `Tool error: sql: <message>`); `:period` binds to the **golden’s** period (not the project’s latest).
  - `submit_inventory` — arguments = a `SubmittedInventory` (§2): the agent proposes `familyId` + `sql` or `reason`; the app decides status. Zod + `validateInventoryAnchors` at the tool boundary; failures return byte-exact tool errors (§11) and the loop continues. A valid submit ends the loop.
- Contract in the prompt: inventory **all narration-driving values** (figures, counts, amounts, percentages, dates that come from data — not styling numbers like “three sections”) and **all tables**; bind each to a family + SQL that reproduces it, using `:period` for periodic families so bindings transfer across periods; probe with `run_sql` before submitting; reuse sibling item ids and SQL patterns wherever the same logical value appears.
- Caps: 16 rounds, then one nudge (“submit your best inventory now”), then null. Null (or thrown client error) → route reports `bind failed: <message>`; the golden keeps its prior inventory if any.
- Re-run with feedback: prior inventory is passed with the amendment instruction — **keep existing item ids and bound SQL for items the feedback does not dispute**.

### 5a. Cross-period sibling reuse

`siblings` = the bound inventories of the blueprint’s other goldens (any kind), each tagged with its period, capped at the 3 most recently bound and compactly rendered (item id, raw, familyId, sql, status — bound items only). The prompt instructs: “These are verified binding patterns from other periods of the same report family. Try these first: the same logical value binds with the same SQL at a different :period.” Because inventory SQL is `:period`-parameterized, reuse is literal. Steady state mirrors parse plans: the first golden costs a full agent round; later periods mostly re-verify known patterns. Sibling context is prompt-level only — no new stored artifact.

## 6. Verification (app-executed, both golden kinds)

`verifyInventory({ inventory, runSql, period }): BindingInventory` (engine, pure given `runSql`) executes every item with a binding and stamps `verifiedValue` + `status`:

- SQL referencing `:period` when the golden has no period → status `error`, reason `golden has no period`.
- SQL execution error → status `error`, reason `sql error: <message>` (first line, 200 chars).
- `value` items: result must be a single row with a single column, else status `error`, reason `sql did not return a single value`. Numeric comparison: match iff `|verified − parsed| ≤ max(0.005, 0.01·|parsed|)` (covers rounded narration like “$4.2M” vs 4,215,332). String comparison: case-insensitive trimmed equality. Match → `bound`; ran but no match → `mismatch` (both values kept and shown). `parsed` null with a binding → `mismatch` never applies; status `bound` iff the SQL returns a non-null scalar (existence binding), else `error`.
- `table` items: SQL must execute; column count must equal the anchored table block’s column count, else `mismatch` with reason `column count 3 ≠ 4` (actual numbers). Row count equal to the block’s row count → `bound`; differing → `mismatch`, reason `row count <got> ≠ <want>`. `verifiedValue` = row count.
- Items with `binding: null` pass through unchanged (unbound, agent-supplied `reason`).

Verification is synchronous and cheap (read-only SQL); it runs after every agent submit, after deterministic inventory builds, and on every “verify bindings” click. **Mismatch is load-bearing**: a golden whose period data was later corrected shows mismatches rather than silently rebinding.

## 7. Triggers and flow

- **Exemplar upload** → extract → unify → (on success) bind → verify → store. One request; the golden card shows the outcome.
- **Star run/section** → deterministic inventory → verify → store.
- **Bind route**: `POST /api/blueprints/[id]/goldens/[goldenId]/bind` body `{ feedback?: string }`. Behavior matrix:
  - 404 unknown golden (or golden not on this blueprint).
  - Exemplar without `document` → 400 `golden is not unified`.
  - Exemplar, no stored inventory OR `feedback` present → agent round (§5) then verify; 500 `bind failed: <message>` on null, prior inventory intact.
  - Exemplar, stored inventory, no feedback → re-verify only (zero LLM).
  - Run/section golden with `feedback` → 400 `feedback requires an exemplar golden`; without → deterministic rebuild (§4) + verify.
- **Retry unify**: `POST /api/blueprints/[id]/goldens/[goldenId]/unify` — re-extracts and re-runs §3, then auto-binds on success. 400 for non-exemplar goldens (`only exemplar goldens can be unified`).
- **Edit period**: `PATCH /api/goldens/[id]` accepts `{ period: string | null }` (canonical form or null, else 400 `invalid period: <value>`) alongside the existing fields; changing it re-runs verification (not the agent).

## 8. Access layer (the abstraction)

`resolveGoldenText` is **retired**. One resolver replaces it, in `apps/web/lib/goldens.ts`:

- `resolveGolden(db, goldenId): Promise<ResolvedGolden | null>` where `ResolvedGolden = { id, kind, label, note, period, document: RunDocument | null, inventory: BindingInventory | null, unifyError: string | null }`. Run/section documents materialize from the run row; exemplar documents from `goldens.document`. `document: null` (failed/pending unify) means the golden is inert: agents see only its label and unify status.
- `renderGoldenForPrompt(g: ResolvedGolden): string` (engine, pure): headings + prose + tables from the document, with binding annotations inline where an anchor is bound — `«$4.2M ← fam_ar_transactions: SELECT SUM(amount)…»` (SQL truncated at 120 chars) — and a trailing `boundness: 7/9 bound, 1 mismatch, 1 unbound` line. This one renderer serves every consumer.
- **Copilot**: golden summaries in the system prompt gain boundness (`7/9 bound`); a new read tool `get_golden { goldenId }` returns `renderGoldenForPrompt` output. Scaffolding needs no new machinery beyond this: the copilot reads a bound golden and lifts its SQL into `update_section_queries` / section prose.
- **Generation**: every current `resolveGoldenText` call site switches to `renderGoldenForPrompt`. Prompt shape may change (structured headings instead of packed text) — the eval battery is the regression net.

## 9. UI

`MemoryGoldenDrawer` golden cards gain (Manuscript idioms, `data-testid={golden-<id>}`):

- Unify badge for exemplars: `unified` / `unify failed` (amber, with `unify_error` text and a “retry unify” button).
- Editable period chip (same editing idiom as source chips).
- Boundness line: `7/9 bound · 1 mismatch · 1 unbound` with a thin progress bar.
- “Bind to data” (exemplar, first run/with feedback) / “verify bindings” (run/section, and exemplar re-verify).
- Expandable inventory: one row per item — raw text, `familyId` + SQL (mono, truncated) when bound; both values in amber on mismatch (`says $4.2M · data 3,981,102`); reason when unbound. Feedback input + “rebind” button (exemplars only).

## 10. Seed and evals

- **Seed**: a markdown exemplar `ar_review_q1_2026.md` **generated by seed** (template in seed.ts, figures interpolated from the warehouse seed just built, written to RUNOFF_FILES_DIR like other stored files — text and data agree by construction; nothing committed under fixtures). The goldens row is inserted with a **baked literal document** (`AR_REVIEW_DOC` const, the `AR_AGING_PLAN` pattern) and an inventory whose bindings seed computes and verifies directly against the warehouse it just built — a fully bound golden out of the box, zero LLM at seed time.
- **`pnpm eval:golden`** (`scripts/evalGolden.ts`, sibling of eval:parse): live smoke — unify the raw markdown fixture (ignoring the baked doc), assert ≥ 2 sections and a detected period `2026-Q1`; then bind with the real agent against the seeded warehouse and assert: the known AR total (derived from the warehouse at eval time, like eval:sql) appears as a `bound` item whose `verifiedValue` matches, and overall boundness ≥ 0.5. Output `EVAL GOLDEN OK` / `EVAL GOLDEN FAIL: <detail>`. package.json entry mirrors siblings.
- **Tests**: all deterministic with mocked clients — unify validation/degeneracy/period rules; anchor validation; verification truth table (tolerance boundaries, single-value rule, table col/row rules, no-period, sql error); deterministic citation inventory incl. compileLocator round-trip; sibling-context-reaches-prompt; amendment keeps ids; route statuses; UI cases (badge, inventory rows, mismatch amber, rebind wiring). Full suite green at every task (additive plan).

## 11. Byte-exact strings

- `unsupported exemplar type for unify: <mime>` · `unify failed: <message>` · `only exemplar goldens can be unified`
- `bind failed: <message>` · `golden is not unified` · `invalid period: <value>`
- Tool errors (bind loop): `Tool error: sql: <message>` (existing) · `Tool error: invalid inventory: <zod-or-anchor message>` where anchor messages are `unknown section: <sectionKey>` · `block index out of range: <sectionKey>[<blockIndex>]` · `span index out of range: <sectionKey>[<blockIndex>][<spanIndex>]` · `anchor kind mismatch: <id>` · `duplicate item id: <id>`
- Verification reasons: `golden has no period` · `sql error: <message>` · `sql did not return a single value` · `column count <got> ≠ <want>` · `row count <got> ≠ <want>` · `no query covers this table`
- Eval: `EVAL GOLDEN OK` · `EVAL GOLDEN FAIL: <detail>`

## 12. Out of scope (deliberate)

Blueprint-level stored “binding map” artifact (sibling inventories are the memory); cross-period value search for period-less goldens; unifying CSV/XLSX exemplars; docx table fidelity beyond what the extractor yields; auto-scaffolding button (“build blueprint from this golden” — copilot can already do it conversationally); binding confidence scores; multi-golden batch rebind.

## 13. Accepted limitations

- The ±1% numeric tolerance can false-positive on coincidentally close values and false-negative on aggressively rounded narration (“~$4M” vs 4,215,332 → mismatch). The mismatch row shows both values, so the user sees why.
- Unify’s verbatim-preservation instruction is a prompt contract, not a guarantee; the raw file is retained and re-unify is cheap.
- Table binding verifies shape (columns, row count), not cell contents.
- Sibling reuse is capped at 3 inventories to bound prompt size; older patterns age out of context (not out of storage).
- `document` for run/section goldens reflects the run as stored; if a run is deleted, the golden resolves with `document: null` (inert) — matching current behavior of dangling run goldens.
