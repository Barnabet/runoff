# Runoff v1.3b — Run Pipeline on the Warehouse (Design)

Approved design, 2026-07-19. Successor to v1.3a (`2026-07-19-runoff-v1.3a-data-plane-design.md`), which built the warehouse, catalog, and copilot `run_sql`. v1.3b moves the run pipeline onto that warehouse: sections carry copilot-baked `:period`-parameterized SQL queries whose results replace the 40-row prompt pack; citation locators compile to SQL and verify against the full data; assert checks are scalar SQL. Four v1.3a deferred fixes ride along.

## 1. Overview

- Drafting prompts feed each section the **results of baked SQL queries** run against the project warehouse for the run's period — not a 40-row head-slice of raw files.
- **Citation locators** keep the compact `agg(table.column [where col=value])` grammar but are **compiled to SQL and executed against the full warehouse** at audit time. The 40-row blindness of v1's verification is gone.
- **Assert checks** store raw scalar SQL (`:period`-parameterized) with an operator and target; the old `expression` grammar field is deleted.
- Sections with queryable families but **no baked queries get auto-default queries** (`SELECT * … LIMIT 40`) — the v1 behavior becomes the degenerate case; nothing breaks on day one.
- Baked queries are **copilot-authored, visible read-only** in the Builder section card. No manual SQL editor.
- The copilot's **`compute` tool is retired** (`run_sql` covers it).
- Document/PDF families keep the existing text-extraction pack path unchanged; they are not queryable.
- Backlog fixes shipped in the same release: ingest-lock widening, CSV NULL ingestion, classify-enrichment isolation, eval:sql sum assertion (§10).

Architecture decision (approved): the engine reaches the warehouse only through an injected `RunData` interface built by the worker (mirrors `CopilotContext.runSql`). The engine never imports `@runoff/core`'s `warehouse.ts`; engine tests inject fakes.

## 2. Scope

**In:** blueprint content schema (sections.queries, assert rule shape); core `runWarehouseSql` named-param binding; engine `RunData` type, prompt data block, locator compiler, SQL checks, dialect prompt text; worker `RunData` construction; copilot `update_section_queries` op + `compute` retirement + baking guidance; Builder query chips; seed baked queries + SQL asserts; evals; the four backlog fixes; retirement of sourcePack's tabular parsing.

**Out (unchanged):** warehouse ingestion pipeline (beyond backlog fixes), classify flow (beyond fix c), catalog builder/serializer, copilot `run_sql`/`query_sources`/memory/goldens, run event *types*, Reader, RunDialog, delivery, distillation.

## 3. Blueprint content schema (breaking — delete-and-reseed)

`packages/core/src/types/blueprint.ts`:

```ts
export const SectionQuerySchema = z.object({
  name: z.string().regex(/^[a-z][a-z0-9_]*$/),
  sql: z.string(),
  description: z.string().optional(),
});

export const RuleSchema = z.object({
  kind: z.enum(["assert", "style", "judgment"]),
  text: z.string(),
  sql: z.string().optional(),        // assert only: scalar SELECT, may use :period
  op: z.enum(["==", "<=", ">=", "<", ">"]).optional(),
  value: z.number().optional(),
  withinPct: z.number().optional(),  // tolerance as % of |value|
});

export const BlueprintSectionSchema = /* existing fields */ .extend({
  queries: z.array(SectionQuerySchema),   // NEW, required (may be [])
});
```

- The old `Rule.expression` field is **deleted**. An assert rule is evaluable iff `sql`, `op`, and `value` are all present; an assert missing any of them fails its check with detail `assert rule is missing sql/op/value`.
- Query `name` is unique within a section (enforced at op-apply time, §7).
- Old blueprint revisions no longer parse → upgrade is **delete `data/runoff.db` + `data/warehouses/` + `pnpm seed`** (openDb guard already names the command). No app-DB DDL change.

## 4. RunData — the injected data boundary

`packages/engine/src/runData.ts` (new):

```ts
export interface RunFamily extends CatalogFamily {
  id: string;   // family id — matches BlueprintSection.familyIds and citation sourceIds
}

export interface RunData {
  catalog: RunFamily[];                 // bound families only, documents included (queryable: false)
  exec(sql: string): SqlResult;         // read-only; throws on SQL error
}
```

- The **worker** builds it (`apps/worker/src/runData.ts` or inline next to `resolveSources.ts`): `catalog(db, projectId)` joined with family ids, filtered to the union of all sections' `familyIds`; `exec` = `runWarehouseSql(projectId, sql, { period })`.
- `executeRun` gains a required `data: RunData` option. `files` stays (document families still flow through the pack).
- Engine tests construct fake `RunData` objects; no SQLite in engine tests.

### Core: named-parameter binding

`runWarehouseSql(projectId, sql, params?: { period?: string | null })` in `packages/core/src/warehouse.ts`:

- The statement is bound with `{ period }` **iff** the SQL text matches `/:period\b/`. (Copilot guidance keeps `:period` out of string literals; a false positive inside a literal is acceptable for a local app.)
- If the SQL references `:period` and `params?.period` is null/undefined, throw byte-exact: `query references :period but no period was provided`.
- All existing behavior (readonly open, `query_only`, single statement, `no data ingested yet`) unchanged.

## 5. Section prompt data block

Replaces `packForPrompt` for **queryable** families in `sectionUserPrompt`. For each bound family, in `familyIds` order:

**Queryable family:**
```
### <label> (<familyId>)
<one line per table: `<tableName>(<col TYPE, …>) — <n,nnn> rows`>
-- <queryName>: <sql, newlines collapsed to single spaces>
<formatSqlResult(result)>
```
- One `-- name: sql` + result block per query. Results serialize with core's existing `formatSqlResult` — same 200-row / 10,000-char caps, `(0 rows)`, and `… truncated at <N> of <M> rows` line as `run_sql`. One serializer everywhere.
- A query whose execution throws renders `-- <name>: <sql>` followed by `query failed: <message>` instead of a result — the section still drafts; the model is told the query failed.
- **Auto-default**: if the section has zero baked queries whose SQL references any of this family's tables, the run synthesizes one default per table, named `default_<tableName>`: `SELECT * FROM "<t>" WHERE _period = :period LIMIT 40` (periodic) / `SELECT * FROM "<t>" LIMIT 40` (constant). Defaults render identically to baked queries. ("References" = the table name appears as a word in the query SQL.)

**Document/PDF family:** exactly today's `packForPrompt` output (`### label (id)`, summary, extracted text ≤ 8,000 chars). `buildSourcePack` now parses **only** document/PDF files; tabular files in `files` are skipped by the pack (their data comes from the warehouse).

The drafting **system prompt's dialect contract** is updated: locators are `agg(tableName.column)` where `tableName` is one of the tables shown in the section's data block, and the cited `sourceId` must be the familyId owning that table. Example in prompt: `[[220,500|fam_ab12|sum(fam_ar_transactions.amount where status=paid)]]`. Wording is otherwise preserved (figures, placeholder ban, quote references for documents).

## 6. Citations & checks

### Locator compiler

`packages/engine/src/checks.ts` — `computeAgg`/`computeLocator` (pack-based) are **deleted**; new `compileLocator(expression, catalog): { sql, table } | null`:

- Grammar unchanged: `^(sum|avg|min|max|count)\((\w+)\.(\w+)(?:\s+where\s+(\w+)\s*=\s*([^)]+?))?\)$` — `\w+` covers warehouse table names (`fam_x`, `fam_x__slug`).
- Compiles to `SELECT <AGG>("<col>") FROM "<table>"` + WHERE clauses joined by AND:
  - `_period = :period` iff the table (per catalog) has a `_period` column;
  - filter, when present: value trimmed, single quotes doubled; if it parses as a finite number `n`: `("<fc>" = <n> OR lower(CAST("<fc>" AS TEXT)) = lower('<v>'))`, else `lower(CAST("<fc>" AS TEXT)) = lower('<v>')` — preserving the old case-insensitive string-compare semantics.
- `count` compiles to `COUNT("<col>")` (non-NULL). Parity with the old non-empty count holds because of backlog fix (b): CSV empty cells now ingest as NULL.

### Citation audit (`auditCitations`)

Same walk, signature takes `RunData` instead of the pack. Per figure span with an aggregate-shaped locator:

1. Resolve the locator's table to the owning family via `catalog` (the family whose `tables` include it). No owner → failure `unverifiable locator: <locator>`.
2. Owner id ≠ `span.citation.sourceId` → failure `locator source mismatch: cites <sourceId> but locator references <table>`.
3. Execute the compiled SQL via `data.exec`. Any throw → `unverifiable locator: <locator>`. Non-single-numeric result → same failure.
4. Compare with the existing 0.5% relative tolerance → `citation mismatch: <fig> vs computed <n>`.

`uncited figure:`, `cited span has no figure:`, `figure cites unbound source` behaviors and strings unchanged. Non-grammar locators remain quote references and are not recomputed (document citations unchanged).

### Assert checks

Per assert rule: execute `rule.sql` via `data.exec`.
- SQL throw (including core's `query references :period …` and `no data ingested yet`) → check fails, detail = the error message.
- Result must be exactly one row, one column, numeric (`typeof === "number" && finite`) → else fail with detail byte-exact: `check query must return one numeric value`.
- Compare with the existing `compare()` semantics (`op`, `value`, `withinPct` as % of |value|; integer-exact / 0.01 float slack on `==`). Detail format: `<sql collapsed to one line> = <actual> (expected <op> <value>[ within <pct>%]) — pass|fail` with `en-US` formatting, mirroring today's shape.
- `check_passed` / `check_failed` events and retry-then-flag flow unchanged.

## 7. Copilot & Builder

- **New edit op** `{ type: "update_section_queries", sectionKey, before: SectionQuery[], after: SectionQuery[] }` — invertible by swapping before/after, mirroring `update_global_rules`. Apply-time validation (web, at op commit): names match `^[a-z][a-z0-9_]*$` and are unique in the section, else reject with `invalid query name: <name>`; each `after` query is executed once via `runWarehouseSql(projectId, sql, { period: <latest filed period across the project, else null> })`, and a throw rejects the whole op with `invalid query <name>: <message>`.
- **`compute` tool deleted**: tool definition, executor case, `activityLabel` entry, and any prompt mention. `run_sql` is the copilot's computation path.
- **Copilot system prompt** gains baking guidance: bake queries with `update_section_queries`; parameterize periodic tables with `:period`; keep results small and purposeful (the drafting model sees them verbatim); name queries like identifiers; asserts are scalar SQL + op + value.
- **Builder section card** gains a read-only "Data queries" block: per query `<name> — <n> rows` (row count from a server-side dry run at payload-build time — `SELECT COUNT(*) FROM (<sql>)` with the same latest-period binding; errors render `invalid` instead of a count) with the SQL shown beneath in small mono; plus a `defaults active for: <family keys>` line when a queryable family has no covering baked query. Manuscript palette; copilot is the only editor.

## 8. Worker & run events

- Worker: build `RunData` next to `resolveRunSources` (needs `projectId` — one join from blueprint), pass to `executeRun`. Nothing else changes; `run_started`, `source_read` (documents only now), stats, flags, distillation untouched.
- No new run-event types. Check details become SQL-flavored strings (Reader renders them as opaque text already).

## 9. Retirements

- `sourcePack.ts`: `buildCsv`, `buildXlsx`, `ParsedTable`, `serializeTable`, `NUMERIC_SUMMARY_COLUMNS`, and table-summary code deleted; the module keeps document/PDF extraction, `packForPrompt` (documents only), `cellText`/`cellValue` (used by `tabular.ts`). `DEFAULT_MAX_ROWS = 40` survives only as the auto-default query LIMIT (moves to wherever defaults are synthesized).
- `checks.ts`: pack-based `computeAgg`/`computeLocator` deleted (grammar regex + `compare()` + audit walk survive).
- `copilot.ts`: `compute` tool fully removed.
- `query_sources`' document path and everything else copilot-side stays as v1.3a shipped it.

## 10. Backlog fixes (from v1.3a final review)

a. **Ingest lock widening**: the upload route's row INSERT + file write, and the source DELETE route (row delete + warehouse cleanup), run inside `withIngestLock`. Retires both v1.3a concurrency findings (interleaved writes in the ingest await gap; DELETE racing an ingest's ATTACH).
b. **CSV NULL ingestion**: empty CSV cells reach `insertRows` as `null` (not `""`), matching XLSX. Applied at the ingest boundary in `tabular.ts`'s CSV row delivery; type inference already treats empties as absent. Warehouse `IS NULL` / `COUNT(col)` semantics become correct. (Dev data: delete-and-reseed.)
c. **Classify enrichment isolation**: in the classify route, `scanTabular` failure on a tabular file falls back to `readContentSample` (classification still happens); enrichment (`tableNamesFor`/`readWarehouseTables`/`computeDrift`) gets its own try — on throw, the un-enriched proposal is kept instead of nulled.
d. **eval:sql assertion**: the eval computes the expected value directly (`runWarehouseSql(projectId, "SELECT SUM(amount) FROM fam_ar_transactions WHERE _period = '2026-Q2'")`), then asserts the model's reply, digits-only-normalized (strip everything but `[0-9]`), contains the expected integer part's digits. Failure string: `EVAL SQL FAIL: reply does not contain expected sum <n>`.

## 11. Seed & evals

- Seed: the AR-bound section gains exactly two baked queries — `total_by_status` (`SELECT status, SUM(amount) AS total FROM fam_ar_transactions WHERE _period = :period GROUP BY status ORDER BY status`) and `invoice_count` (`SELECT COUNT(*) FROM fam_ar_transactions WHERE _period = :period`) — and one SQL assert (`sql: SELECT SUM(amount) FROM fam_ar_transactions WHERE _period = :period`, `op: ">"`, `value: 0`); other sections keep `queries: []` to exercise the auto-default path. Asserts elsewhere are rewritten from the old grammar to scalar SQL.
- `scripts/eval.ts` calls `executeRun` directly, so it builds `RunData` the same way the worker does (shared helper or mirrored construction).
- `pnpm eval` continues to assert EVAL OK end to end and gains an assertion that at least one `check_passed` fired (proving SQL checks executed).
- `pnpm eval:copilot` unchanged; `pnpm eval:classify` unchanged; `pnpm eval:sql` strengthened per §10d.
- Full live battery re-verified before merge.

- **Web**: op validation (bad name, duplicate name, invalid SQL, valid apply + invert); Builder chips render (jsdom); classify fallback + enrichment isolation; ingest-lock widening (upload/DELETE during a stubbed slow ingest).
