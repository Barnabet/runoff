# Runoff v1.2b — Projects, Smart Source Manager, Project Memory

**Approved:** 2026-07-18 (brainstormed with Louis).
**Decomposition:** second half of the v1.2 vision. v1.2a (builder copilot + blueprint-scoped memory + goldens) shipped to main earlier today; this spec builds the organizational level above it.
**Decisions locked during brainstorm:** projects own blueprints AND sources; sources are agent-organized into a `family/period` taxonomy with classify-then-confirm at upload; one live file per family per period; family kind is `periodic` or `constant`; one fixed granularity per periodic family, no mixing — and all periodic families bound to a blueprint must share one granularity; blueprints bind families (file-ID binding retired, no pinning escape hatch); the run period is chosen at trigger time with per-family gaps shown, proceeding with gaps allowed; memory gains a project scope with the agent (copilot + distiller) deciding scope; Home becomes the project list and the global `/sources` page is deleted; **no schema migration** — breaking change, delete-and-reseed is the upgrade path (only fixture data exists).

Out of scope (deferred): cross-project shared source pool; multi-file periods; file-ID pinning; memory retrieval/embeddings (inject-all with caps stands); moving blueprints between projects; project archival/deletion UI beyond create+rename.

---

## 1. Data model (`@runoff/core`)

Breaking schema change in the boot DDL (`packages/core/src/db/index.ts`) and drizzle `schema.ts`. No `ALTER`/migration machinery: the DDL is rewritten in place with the new shapes. `openDb` gains a guard — after running the DDL, it checks `SELECT COUNT(*) FROM pragma_table_info('sources') WHERE name='family_id'`; if the column is missing (pre-v1.2b database file), it throws `Error("database predates v1.2b — delete the DB file and run: pnpm seed")`. `pnpm seed` recreates everything (§7).

```sql
CREATE TABLE IF NOT EXISTS projects (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE IF NOT EXISTS source_families (
  id TEXT PRIMARY KEY,
  project_id TEXT NOT NULL,
  key TEXT NOT NULL,              -- slug, e.g. 'trade_data'; unique per project
  label TEXT NOT NULL,            -- display, e.g. 'Trade data'
  kind TEXT NOT NULL,             -- 'periodic' | 'constant'
  granularity TEXT,               -- 'quarter' | 'month' | 'year'; NULL iff kind='constant'
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  UNIQUE(project_id, key)
);
```

`sources` is rebuilt (same table name):

```sql
CREATE TABLE IF NOT EXISTS sources (
  id TEXT PRIMARY KEY,
  project_id TEXT NOT NULL,
  family_id TEXT,                 -- NULL while unfiled
  period TEXT,                    -- canonical string; NULL for constant families and unfiled rows
  name TEXT NOT NULL,             -- original filename
  kind TEXT NOT NULL DEFAULT 'file',
  stored_filename TEXT NOT NULL,
  mime TEXT NOT NULL,
  size INTEGER NOT NULL,
  status TEXT NOT NULL DEFAULT 'unfiled',  -- 'unfiled' | 'filed' | 'replaced'
  proposal TEXT,                  -- JSON ClassifyProposal from the classifier; NULL if classification failed
  uploaded_at TEXT NOT NULL DEFAULT (datetime('now')),
  filed_at TEXT
);
-- one live file per periodic slot / per constant family:
CREATE UNIQUE INDEX IF NOT EXISTS sources_slot ON sources(family_id, period) WHERE status='filed';
-- (period IS NULL for constant families; SQLite treats NULLs as distinct in unique
--  indexes, so constant one-live-file is enforced in code at confirm time, not by this index.)
```

Other table changes:

- `blueprints` gains `project_id TEXT NOT NULL`.
- `blueprint_sources` is **dropped**, replaced by `blueprint_families(blueprint_id TEXT NOT NULL, family_id TEXT NOT NULL, PRIMARY KEY(blueprint_id, family_id))`.
- `runs` gains `period TEXT` (NULL when the blueprint binds no periodic families).
- `memories` gains `scope TEXT NOT NULL DEFAULT 'blueprint'` (`'blueprint' | 'project'`) and `project_id TEXT NOT NULL` (set on every row — for blueprint-scoped rows it is the owning blueprint's project, making per-project queries trivial). `blueprint_id` becomes nullable; NULL iff `scope='project'`.

### Periods and granularity

A period is a canonical string, validated by regex per granularity:

| granularity | format | regex | display |
|---|---|---|---|
| `quarter` | `2026-Q1` | `^\d{4}-Q[1-4]$` | `Q1 2026` |
| `month` | `2026-06` | `^\d{4}-(0[1-9]\|1[0-2])$` | `Jun 2026` |
| `year` | `2026` | `^\d{4}$` | `2026` |

Within one granularity, lexicographic order is chronological, so "latest period" is `MAX(period)`. Mixing never occurs: a family has exactly one granularity, and a blueprint may only bind periodic families sharing one granularity (enforced at bind time and re-validated at run trigger, §4).

Core exports (in `packages/core/src/types/sources.ts`, new): `FamilyKind`, `Granularity`, `PERIOD_REGEX: Record<Granularity, RegExp>`, `formatPeriod(period, granularity): string`, `ClassifyProposalSchema` (zod, §3).

### Blueprint content change

In `BlueprintContentSchema` (`packages/core/src/types/blueprint.ts`), the per-section field `sourceIds: string[]` is **renamed to `familyIds: string[]`** (family IDs). The rename propagates through the reducer, diff, EditOps (`edit_section` patches may carry `familyIds`), engine prompts, and all web components. There is no back-compat parsing: old revisions' JSON fails validation, which is acceptable under the delete-and-reseed upgrade path.

Files on disk stay exactly as today: opaque `stored_filename` blobs under `RUNOFF_FILES_DIR`. The `project/family/period` path is a *virtual* taxonomy in SQLite, never a directory layout.

## 3. Smart source manager

### Upload → classify → confirm

1. **Upload** (`POST /api/projects/[id]/sources`, multipart, multiple files): each file is stored to `RUNOFF_FILES_DIR` immediately and inserted as `status='unfiled'`; the response returns the new rows without waiting on any LLM call.
2. **Classify** (`POST /api/projects/[id]/sources/classify`, `{sourceIds}`): called by the client right after upload (and re-callable from a "reclassify" affordance on a chip). The route calls the engine's `classifySource` once per file (sequentially) and persists each result into `sources.proposal`, returning the updated rows; the tray shows `classifying…` on a chip until its proposal lands. Classification failure (proxy down, unparseable reply) leaves `proposal` NULL — the row still appears in the unfiled tray with empty pickers. Classification never blocks or fails an upload.
3. **Confirm**: the unfiled tray on the project page shows each row as an editable proposal chip — `trade_data · Q2 2026 (high)` — with pickers to change family (existing families + "new family…" with key/label/kind/granularity fields) and period (validated live against the family's granularity regex). "Confirm" (per row) and "Confirm all" (rows with a proposal) call `POST /api/projects/[id]/sources/confirm` with `{sourceId, familyId | newFamily, period}`.
4. **Filing** (in the confirm route, transactional): create the family if `newFamily`; validate period against granularity (constant ⇒ period must be null); if a `filed` row already occupies the slot (same family+period, or same constant family), mark it `status='replaced'`; set the confirmed row `family_id`, `period`, `status='filed'`, `filed_at`. The UI warns "replaces <name>" before confirming into an occupied slot.

Refiling later (wrong period, wrong family) is `PATCH /api/projects/[id]/sources/[sourceId]` with the same `{familyId, period}` shape and the same slot/replace rules. `DELETE` removes an unfiled or filed row (filed deletion frees the slot; the stored file is deleted from disk).

### Engine: `classifySource`

New `packages/engine/src/classify.ts`:

```ts
export async function classifySource(opts: {
  client: OpenAI; model: string;
  filename: string;
  contentSample: string;          // first 2,000 chars of the parsed file
  families: { key: string; label: string; kind: FamilyKind; granularity: Granularity | null }[];
}): Promise<ClassifyProposal | null>
```

One chat call, JSON-object response, zod-validated against `ClassifyProposalSchema`:

```ts
{
  familyKey: string;                    // existing key, or the proposed new key
  newFamily?: { key: string; label: string; kind: FamilyKind; granularity: Granularity | null };
  period: string | null;                // null iff the (new or existing) family is constant
  confidence: 'high' | 'medium' | 'low';
}
```

Validation after parse: if `familyKey` matches an existing family, `newFamily` must be absent and `period` must match that family's granularity (or be null for constant); if new, `newFamily` is required and `period` must match `newFamily.granularity`. Any violation ⇒ return `null` (treated as classification failure). The prompt lists existing families with their kinds/granularities and instructs: prefer existing families; propose `constant` only for clearly non-periodic reference material; emit periods in canonical form.

### Family tree display

Below the unfiled tray, the manager lists families: constant families show their single live file; periodic families show a period row (`Q1 2026 ✓ · Q2 2026 ✓ · Q3 2026 —`) ordered ascending, rendering gaps between the family's min and max period explicitly. Each filed entry offers refile/delete.

## 4. Family binding and period-resolved runs

### Binding

The Builder's ContentsRail SOURCES area switches from per-file checkboxes to per-family checkboxes (project families, showing kind and granularity). This UI is also rebuilt to fix the v1.2a discoverability flaw: an explicit italic `+ bind sources…` affordance (matching the `+ add a section…` idiom) replaces the invisible heading-as-toggle. Binding writes `blueprint_families` immediately via `PATCH /api/blueprints/[id]` (`{familyIds}`), replacing today's `{sourceIds}` behavior. The API rejects (400) a set whose periodic families span more than one granularity; the UI disables incompatible checkboxes with a `— granularity differs` hint once a first periodic family is ticked. Per-section `familyIds` (§1) select among the blueprint's bound families, exactly as sections selected files before.

### Run trigger

`GET /api/blueprints/[id]/run-options` returns:

```ts
{
  granularity: Granularity | null;      // null if no periodic families bound
  periods: { period: string; families: { key: string; label: string; present: boolean }[] }[],
                                        // one row per period that ANY bound periodic family has filed,
                                        // descending; families = every bound periodic family
  constants: { key: string; label: string; present: boolean }[]
}
```

The Run button opens a small dialog: period select defaulted to the latest period with any data, the per-family checklist for the chosen period (`✓ trade_data · ✗ spend_data — missing`), and constant families listed with their presence. Proceeding with gaps is allowed — one confirm click. Blueprints with only constant families (or none) skip the period select and show only the checklist. `POST /api/runs` gains `period: string | null`, validated against `run-options` (must be a listed period, or null when `granularity` is null).

### Resolution (worker)

New pure function in `apps/worker` (or `packages/engine`): `resolveRunSources(db, blueprintId, period)` → `{ fileSourceIds: string[], gaps: { familyKey: string }[] }`: for each bound family, periodic ⇒ the `filed` row at `period` (absent ⇒ gap), constant ⇒ its live `filed` row (absent ⇒ gap). `processOne` calls it, builds the sourcePack from the resolved file rows **through the existing sourcePack pipeline unchanged**, and records the run's `period`. Per-section `familyIds` filter the pack the same way per-section `sourceIds` did. `run_started`'s payload gains `period` (spread-omitted when null, preserving the v1.1 byte-compat idiom alongside `memoryIds`) and its existing source listing reflects the resolved files; gaps are recorded in the payload as `gaps: string[]` (family keys, spread-omitted when empty).

Run history rows, the Live Run header, and the Reader masthead display the period (via `formatPeriod`). Run-over-run continuity is untouched: `previousCompletedDocument` remains the blueprint's latest completed run regardless of period.

## 5. Project memory

- **Writing.** The copilot `save_memory` tool gains a required `scope` parameter (`'blueprint' | 'project'`), described as: blueprint = about this document's content/structure; project = about the client or its data, true for every document in the project. `distillRun` output items become `{ body, scope }` with the same instruction in its prompt. `saveMemory` in the web/worker query layer takes the scope and writes `project_id` (always) + `blueprint_id` (blueprint scope only).
- **Caps.** Unchanged 30-active per blueprint; new 30-active per project scope (oldest active disabled first). Same never-block failure isolation as v1.2a.
- **Prompt injection** (`prompts.ts` systemPrompt only, and the copilot system prompt's equivalent block): project block first, blueprint block second, each with the v1.2a `- <memory>` line format:
  - Project: `Standing guidance for this project (applies to every document in this project — follow unless blueprint guidance or a section instruction contradicts it):`
  - Blueprint (unchanged wording): `Standing guidance for this blueprint (learned from the builder and past runs — follow unless a section instruction contradicts it):`
  - Either block is omitted entirely when its scope has no active memories.
- **`run_started.memoryIds`** includes both scopes' active memory IDs (project first); still spread-omitted when empty.
- **UI.** The v1.2a memory drawer shows both scopes with `PROJECT` / `BLUEPRINT` mono badges; the project page's memory zone lists project-scoped memories with the same disable/enable. `GET /api/blueprints/[id]/memories` returns both scopes (badged); project memory CRUD reuses `/api/memories/[id]` (disable/enable) plus the project payload for listing.

## 6. Copilot taxonomy awareness

- `buildCopilotContext` gains the project's family tree (families with kind, granularity, and filed periods) so the copilot can answer "what data do we have for Q2?" without a tool call.
- `query_sources` and `compute` become family/period-addressed: `query_sources({ familyKey?, period? })` summarizes bound families (with their filed periods) or inspects one family's file at a period (default: latest; constant families ignore `period`); `compute` takes `familyKey` + optional `period` in place of `sourceId`. Both resolve through the same slot lookup as the worker.
- `edit_section` ops patch `familyIds` (renamed from `sourceIds`); the op may only reference the blueprint's bound families — violations are rejected at the existing validation chokepoint before `commit()`.
- `save_memory` scope per §5. All other tools unchanged.

## 7. Seed, eval, and the upgrade path

- `pnpm seed` rebuilds: one project ("Meridian Advisory" — any plausible fixture name), the existing demo blueprints assigned to it, families derived from the current fixture files (at least: one quarterly periodic family with **two** filed periods, one constant family), `blueprint_families` bindings, and section `familyIds`. Fixture data files may be duplicated/renamed as needed to populate two periods.
- `pnpm eval` triggers its run with an explicit period (the latest seeded one) and asserts the run's `period` round-trips into `run_started`. `pnpm eval:copilot` is updated for the renamed tools/fields. A new lightweight `pnpm eval:classify` smoke calls `classifySource` against a fixture file and asserts a valid proposal (live proxy required, same pattern as `eval:copilot`).
- Upgrade path: delete the DB file, `pnpm seed`. The `openDb` guard (§1) makes the failure mode explicit rather than a crash deep in a query.

## 8. Error handling summary

| Failure | Behavior |
|---|---|
| Classification LLM call fails / invalid JSON / invalid proposal | `proposal` stays NULL; row sits in unfiled tray with manual pickers; upload unaffected |
| Confirm into an occupied slot | prior file marked `replaced` (never deleted from disk); UI warns first |
| Period fails granularity regex at confirm/refile | 400; chip shows inline error |
| Bind set mixes granularities | 400 at PATCH; UI prevents via disabled checkboxes |
| Run triggered with gaps | allowed; gaps recorded in `run_started`; sections lacking data rely on existing audit/unverifiable machinery |
| Run POST with period not in run-options | 400 |
| `save_memory` / distiller failure | never affects the turn/run (v1.2a isolation unchanged) |
| Pre-v1.2b DB file opened | explicit `openDb` error naming the reseed command |
