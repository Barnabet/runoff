# Runoff v1.2a — Builder Copilot + Memory-lite

**Approved:** 2026-07-18 (brainstormed with Louis).
**Decomposition:** v1.2 ships as sub-projects. This spec is **v1.2a**: the builder copilot backed by a minimal blueprint-scoped memory. **v1.2b** (separate spec, later): projects as an organizational level + project-scoped memory, generalizing what 1.2a proves out.
**Decisions locked during brainstorm:** copilot replaces the margin-notes rail; edits apply directly to the Builder draft with per-edit revert (DB untouched until Save); goldens are both starred runs/sections and uploaded exemplars; memories are written by a copilot tool and by a post-run distiller; all memories are active immediately with easy prune; the loop runs in-process in a Next route streaming SSE (approach A).

Out of scope (deferred): projects and project-scoped memory; memory retrieval/embeddings (inject-all with cap); copilot-triggered preview runs; an exemplar-to-blueprint bootstrap wizard.

---

## 1. Data model (`@runoff/core`)

Three new tables in the boot DDL (`packages/core/src/db/index.ts`) and drizzle schema (`schema.ts`). No changes to existing tables; no migration of existing data.

```sql
CREATE TABLE IF NOT EXISTS copilot_messages (
  id TEXT PRIMARY KEY,
  blueprint_id TEXT NOT NULL,
  role TEXT NOT NULL,             -- 'user' | 'assistant'
  body TEXT NOT NULL,
  actions TEXT,                   -- JSON CopilotAction[] (assistant rows only)
  status TEXT NOT NULL DEFAULT 'ok',  -- 'ok' | 'failed'
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE IF NOT EXISTS memories (
  id TEXT PRIMARY KEY,
  blueprint_id TEXT NOT NULL,
  body TEXT NOT NULL,
  source TEXT NOT NULL,           -- 'copilot' | 'distilled'
  origin_id TEXT,                 -- copilot_messages.id or runs.id
  status TEXT NOT NULL DEFAULT 'active',  -- 'active' | 'disabled'
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE IF NOT EXISTS goldens (
  id TEXT PRIMARY KEY,
  blueprint_id TEXT NOT NULL,
  kind TEXT NOT NULL,             -- 'run' | 'section' | 'exemplar'
  run_id TEXT,                    -- kind run/section
  section_key TEXT,               -- kind section
  name TEXT,                      -- kind exemplar: display name
  mime TEXT,                      -- kind exemplar
  stored_filename TEXT,           -- kind exemplar: file in RUNOFF_FILES_DIR
  note TEXT,                      -- optional user annotation, any kind
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

Shared types (new file `packages/core/src/types/copilot.ts`, exported from the core index):

```ts
export type CopilotAction =
  | { kind: "tool"; tool: string; label: string }                    // activity chip, e.g. "reading spend.csv"
  | { kind: "edit"; op: EditOp }
  | { kind: "memory"; memoryId: string; body: string };

export type EditOp =
  | { type: "edit_section"; key: string; before: Partial<BlueprintSection>; after: Partial<BlueprintSection> }
  | { type: "add_section"; afterKey: string | null; section: BlueprintSection }
  | { type: "remove_section"; afterKey: string | null; removed: BlueprintSection }
  | { type: "update_masthead"; before: MastheadPatch; after: MastheadPatch }
  | { type: "update_global_rules"; before: string[]; after: string[] };

export interface MastheadPatch { title?: string; clientName?: string; eyebrow?: string; dateline?: string }

export interface MemoryRow {
  id: string; blueprintId: string; body: string;
  source: "copilot" | "distilled"; originId: string | null;
  status: "active" | "disabled"; createdAt: string;
}
```

`EditOp` carries full before/after values so the client can apply, render a field diff, and revert without another server round-trip. `add_section` and `remove_section` both carry `afterKey` (key of the preceding section, null = end/start-anchor) so each is exactly the other's inverse.

**Margin notes retirement:** delete `MarginNotes.tsx`, the notes routes (`GET/POST /api/blueprints/:id/notes`, `POST /api/notes/:id/accept`, `POST /api/notes/:id/resolve`), and `packages/engine/src/notesAgent.ts` plus their tests. The `notes` DDL and existing rows stay in the DB untouched (readable via SQL; no UI).

## 2. Copilot engine (`packages/engine/src/copilot.ts`)

The interactive sibling of `executeRun`: a bounded streaming tool loop.

```ts
export interface CopilotIO {
  emit(e: CopilotEvent): void;   // streamed to the client as SSE
}
export type CopilotEvent =
  | { type: "text_delta"; text: string }
  | { type: "tool_activity"; label: string }
  | { type: "edit"; op: EditOp }
  | { type: "memory_saved"; memoryId: string; body: string }
  | { type: "done"; messageId: string }
  | { type: "error"; message: string };

export async function copilotTurn(opts: {
  client: OpenAI;
  draft: BlueprintContent;            // the Builder's current (possibly unsaved) draft
  selectedKey: string | null;
  message: string;                    // the new user message
  thread: { role: "user" | "assistant"; body: string }[];  // prior turns, oldest first
  memories: string[];                 // active memory bodies
  ctx: CopilotContext;                // server-side data access, below
  io: CopilotIO;
}): Promise<{ reply: string; actions: CopilotAction[]; draft: BlueprintContent }>;

export interface CopilotContext {
  files: EngineFile[];                                     // bound sources (for the source pack)
  listRuns(): RunSummary[];                                // id, createdAt, status, stats, flag count, rev
  getRunSection(runId: string, key: string): RunSectionDetail | null;  // text + failures/steers/answers/flags for that section
  listGoldens(): GoldenSummary[];
  getGolden(id: string): { description: string; text: string } | null; // run-section text or parsed exemplar text
  saveMemory(body: string): string;                        // returns memory id; enforces caps (§5)
}
```

The loop: system prompt (copilot role, the blueprint-content JSON schema, the draft, selected section, memories block, tool guidance) + thread + new message → `chat.completions.create` with tools, streaming. Text deltas stream as `text_delta`. Tool calls execute server-side, results feed back, loop continues until a final text-only response or **12 tool iterations** (then the loop injects a "wrap up now" system nudge and takes the next text as final).

**Tools** (strict function schemas, like `draft.ts`):

| Tool | Args | Behavior |
|---|---|---|
| `edit_section` | `key`, `patch` (any of heading, mode, instruction, fixedText, sourceIds, rules, number) | Validates patched section against `BlueprintSectionSchema`; applies to the server draft copy; emits `edit` op with per-field before/after (only fields present in patch). Unknown key → tool error result. |
| `add_section` | `afterKey?`, `section` (full BlueprintSection minus number) | Validates; assigns `number` from position and renumbers subsequent sections (renumbering rides the same op); emits op. Duplicate key → tool error. |
| `remove_section` | `key` | Removes + renumbers; emits op with the removed section for revert. |
| `update_masthead` | patch of title/clientName/eyebrow/dateline | Validates non-empty title; emits op. |
| `update_global_rules` | `rules: string[]` | Replaces the list; emits op with before/after. |
| `query_sources` | `sourceId?` | No arg: per-source summaries (name, kind, table columns/row counts, doc excerpt head) from the source pack. With arg: full table summary + first 20 rows serialized (reuse `sourcePack` serializers). |
| `compute` | `expression` | Evaluates one aggregate locator expression — `agg(src.col)` / `agg(src.col where col=value)` — via the same parser/`computeAgg` the audit uses (export what's needed from `checks.ts`). Returns the number or the error text. |
| `list_runs` | — | `ctx.listRuns()`, most recent 10: id, date, status, stats (words, citations, checks, flags, retries). |
| `get_run_section` | `runId`, `key` | `ctx.getRunSection`: the section's document text plus that section's check failures, retry reasons, steers, Q&A, and flag resolutions pulled from `run_events`/`flags`. |
| `list_goldens` / `get_golden` | — / `id` | Summaries; full text (run-section via stored document; exemplar parsed from file — csv/xlsx via source-pack table serialization, pdf/docx/txt via the existing text extraction). |
| `save_memory` | `body` | `ctx.saveMemory`; emits `memory_saved`. |

Every executed tool first emits `tool_activity` with a human label ("reading spend.csv", "checking run of Jul 12", "computing sum(spend.amount)").

The turn's return value includes the final server-side `draft` (with all ops applied) — used by tests and available to the route; the client maintains its own copy by applying streamed ops.

## 3. API + streaming (`apps/web`)

- `GET /api/blueprints/[id]/copilot` → `{ messages: CopilotMessageRow[] }` (thread, oldest first, actions parsed).
- `POST /api/blueprints/[id]/copilot` — body `{ message: string, draft: BlueprintContent, selectedKey: string | null }`. Validates draft with `BlueprintContentSchema` (400 on failure). Persists the user message, builds `CopilotContext` from the DB (bound sources like the worker does, runs, goldens, memories), calls `copilotTurn` with `getLlmClient()`, and streams `CopilotEvent`s as SSE (same `ReadableStream` pattern as the run events route). On completion persists the assistant row (`body`, `actions`, `status:'ok'`) and emits `done` with its id. On error mid-stream: emit `error`, persist the assistant row with `status:'failed'` and whatever text/actions accumulated.
- Memory management: `GET /api/blueprints/[id]/memories` (all rows), `PATCH /api/memories/[id]` (`{status}` toggle), `DELETE /api/memories/[id]`.
- Goldens: `GET /api/blueprints/[id]/goldens`; `POST /api/blueprints/[id]/goldens` — either `{kind:'run'|'section', runId, sectionKey?, note?}` or multipart exemplar upload (same storage pattern as sources upload; mime whitelist reused); `DELETE /api/goldens/[id]`.

The distiller (§5) runs in the worker, not in web.

## 4. Builder UI (`apps/web/components/builder`)

**CopilotRail.tsx** replaces `MarginNotes.tsx` in `Builder.tsx`'s right rail (Manuscript styling: mono labels, hairlines, serif body):
- Thread from `GET .../copilot`, streaming assistant turn rendered live (text deltas; activity chips as small mono lines while tools run).
- **Edit cards**: each `edit` event renders a card inside the assistant message — op title (`§3 channel-performance · instruction`), compact field diff (before in pencil strike/muted, after in ink), and a **Revert** button. Revert applies the inverse op to the draft client-side and marks the card reverted (no server call; it's draft state).
- Composer at the bottom; the selected section shows as a context chip on the composer ("§3 channel-performance") — informational, sent as `selectedKey`.
- A failed turn shows a retry affordance (re-POSTs the same message).
- **Drawer** (toggle in the rail header): two stacked panels — **Memory** (each row: body, source tag `copilot`/`distilled`, active toggle, delete) and **Goldens** (starred runs/sections with dates, exemplar upload button, delete).

**Draft integration in `Builder.tsx`:** applying an op = pure function `applyEditOp(content, op): BlueprintContent` (new file `apps/web/lib/editOps.ts`, unit-tested; inverse via `invertEditOp(op): EditOp`). Ops set the existing `dirty` flag — the Save pill and Cmd/S flow are unchanged, and Save still writes one revision of the whole draft. Fields changed by the most recent assistant turn get a brief highlight in `SectionEditor` (a `copilotTouched: Set<string>` of `${sectionKey}.${field}` passed down; cleared on next user edit or new turn).

**Reader stars:** in `ReaderView`, a star toggle on the run report card (`kind:'run'`) and one per section header (`kind:'section'`), POSTing to the goldens API. Starred state loads client-side via `GET /api/blueprints/[id]/goldens` (the payload shape is untouched); stars are `no-print`.

## 5. Memory-lite

- **Injection into runs:** `executeRun` opts gain `memories?: string[]`. When non-empty, `systemPrompt` (one place — not the per-section prompts) gains the block:

```
Standing guidance for this blueprint (learned from the builder and past runs — follow unless a section instruction contradicts it):
- <memory 1>
- <memory 2>
```

- The worker loads active memories for the blueprint at claim time and passes them; their ids ride `run_started` as `memoryIds?: string[]` (additive event field; reducer stores it; the Reader's run report shows "MEMORY — N standing notes" with the bodies in a tooltip/expandable). `scripts/eval.ts` passes none.
- **Injection into the copilot:** active memory bodies render in the copilot system prompt under a "Standing guidance" heading.
- **Caps:** max **30 active** memories per blueprint, each ≤ 500 chars (`saveMemory` truncates body, and when the cap is hit it disables the oldest active row to make room — never blocks the save). The distiller respects the same caps.
- **Post-run distiller** (`packages/engine/src/distill.ts`, called from the worker after the `run_completed` update): only runs when the run had at least one steer, answer, or resolved flag. One non-streaming LLM call: input = blueprint title + section headings + the run's steers, Q&A pairs, and flag resolutions; output (strict JSON schema) = `{ memories: string[] }` (0–3), each a durable, generalized preference ("Always express spend deltas in percentages"), explicitly instructed to skip one-off facts. Dedupe: skip a candidate whose body case-insensitively equals an existing memory (active or disabled). Insert as `source:'distilled'`, `origin_id` = run id, active. Any distiller error is logged and swallowed — it never affects run status.

## 6. Error handling & guardrails

- Tool-call failures (unknown section key, invalid patch, unparseable expression, missing run) return as tool-result error strings; the loop continues — the model corrects or explains. Nothing thrown crosses the turn boundary except client/transport errors.
- Edit ops are only emitted after zod validation passes; the client applies ops without re-validating (server is authoritative for op shape).
- SSE drop mid-turn: route's finally-block persists what accumulated with `status:'failed'`; the rail shows the partial turn with retry.
- Draft race: the draft the copilot edited is the one the client sent; the client applies ops on top of its live draft. Concurrent user typing during a streaming turn is allowed — ops patch only the fields they name (`edit_section` patches, not whole-section replaces), which keeps clobber risk to the field level; same-field last-write-wins is accepted for v1.2a.
- The 12-iteration loop cap plus per-tool result truncation (tool results ≤ 8,000 chars, matching the source-pack doc cap) bound token spend per turn.

## Global constraints

- No changes to existing tables; new tables via boot DDL only.
- The "Standing guidance" prompt block wording is pinned as written in §5.
- Client components must not import the `@runoff/core` barrel (better-sqlite3 leak) — deep-import or use type-only imports, per existing precedent.
- Printed documents unchanged (Reader stars are `no-print`).
- All existing tests keep passing; `pnpm build` in apps/web stays green; `pnpm eval` unchanged.
