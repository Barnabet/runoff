# Runoff v1.4 — Parse Plans (LLM-in-the-loop source loading)

Approved design, 2026-07-19. Follows v1.3b (run pipeline on the warehouse, main `70ef595`).

## 1. Goal

Turn messy tabular files into correct warehouse tables. Today's ingestion is deterministic but rigid: one header row, blank-line island detection, positional table identity, value-shape typing. Real exports defeat it silently — glued title rows corrupt headers, Grand Total rows poison SUMs, renamed sheets/columns split families, wide layouts stay unaggregatable, formatted numbers land as TEXT.

v1.4 adds a **ParsePlan**: a persistent, zod-typed, family-level artifact that an LLM proposes and deterministic code executes. Bake once on first upload; later periods execute the stored plan with zero LLM involvement; when a file no longer fits, the LLM amends the working plan with execution results in hand. Same contract philosophy as baked section queries: LLM authors, code executes, human confirms.

## 2. Scope

In scope (all four mess classes):
- **Structure cleanup** — glued titles, total/subtotal rows, repeated page headers, footer junk.
- **Stable identity** — table anchoring by header signature (not sheet name/position); column mapping onto canonical names across periods.
- **Reshaping** — multi-row (2–3) headers; unpivot of wide layouts into long key/value rows.
- **Value normalization** — currency/percent/number/date coercion; **validate-only** period derivation (flag rows whose date-derived period mismatches the slot period; optional exclusion; NO multi-period fan-out).

Non-goals (backlog): multi-period fan-out from one file; copilot access to plans; `.xls`/JSON/Parquet; PDF table extraction; streaming plan-path CSV (grids load fully in memory, bounded by the 100MB cap); manual plan editing UI (LLM-authored only, human confirms or gives feedback).

## 3. The ParsePlan artifact

Stored as JSON in new nullable column `source_families.parse_plan`. Core zod schema (`packages/core/src/types/parsePlan.ts`):

```ts
const NAME_RE = /^[a-z][a-z0-9_]*$/;

export const RowRuleSchema = z.object({
  column: z.string().nullable(),      // canonical column name, or null = any cell in the row
  pattern: z.string(),                // case-insensitive regex tested against the cell's raw text
});

export const ColumnPlanSchema = z.object({
  from: z.string(),                   // normalized merged raw header text this column matches
  name: z.string().regex(NAME_RE),    // canonical name; never "_period"
  type: z.enum(["TEXT", "INTEGER", "REAL"]),
  parse: z.enum(["number", "currency", "percent", "date"]).optional(),
});

export const TablePlanSchema = z.object({
  name: z.string().regex(NAME_RE),    // logical table name
  anchor: z.object({
    sheet: z.string().optional(),     // slugified sheet-name HINT only; never required to match
    headerSignature: z.array(z.string()).min(1),  // normalized raw texts of the header row's cells
    minMatch: z.number().int().min(1),
  }),
  headerRows: z.number().int().min(1).max(3),
  exclude: z.array(RowRuleSchema),
  columns: z.array(ColumnPlanSchema).min(1),
  unpivot: z.object({
    keep: z.array(z.string()),        // canonical names that stay as identity columns
    valuePattern: z.string(),         // case-insensitive regex; matched raw headers melt into rows
    keyColumn: z.string().regex(NAME_RE),
    valueColumn: z.string().regex(NAME_RE),
    valueType: z.enum(["INTEGER", "REAL", "TEXT"]),
    valueParse: z.enum(["number", "currency", "percent"]).optional(),
  }).optional(),
  periodColumn: z.string().optional(),          // canonical column (parse: "date") to validate against the slot period
  onPeriodMismatch: z.enum(["keep", "exclude"]).default("keep"),
});

export const ParsePlanSchema = z.object({
  version: z.literal(1),
  tables: z.array(TablePlanSchema).min(1),
});
```

Validity beyond zod (checked by `validateParsePlan`, throws with the detail line): table names unique; column canonical names AND `from` texts unique per table; `unpivot.keep`/`periodColumn`/`exclude[].column` reference declared canonical names; `periodColumn` requires that column's `parse === "date"`.

Warehouse naming: `plan.tables.length === 1` → `fam_<key>`; else `fam_<key>__<name>` — mirrors today's `tableNamesFor` so single-table families keep their table name across the upgrade.

## 4. Executor (`packages/engine/src/parsePlan.ts`)

Pure and deterministic. New shared loader `loadGrids(path, mime, name): Promise<{ sheet: string; grid: unknown[][] }[]>` — XLSX via the existing streaming reader (grids in memory, as today); CSV parses to one grid with `dynamicTyping: false` (raw strings — the plan owns typing; `"007"` stays TEXT unless mapped `parse: "number"`).

`executeParsePlan(grids, plan, slotPeriod, granularity) → { tables: ExecTable[]; report: ExecReport }`.

**Cell normalization** (used for signatures, header matching, rule matching): `String(v).trim().toLowerCase().replace(/\s+/g, " ")`; null/undefined → `""`.

**Anchoring.** Tables resolve in plan order. Scan sheets in file order, rows top-down: a row anchors a table when ≥ `minMatch` of `headerSignature` entries equal the row's normalized cells (positions irrelevant, duplicates count once). First matching row wins; a row consumed as another table's anchor or lying inside another table's resolved region is skipped. `anchor.sheet` only breaks ties (prefer the hinted sheet); it never causes a failure.

**Region.** Header = anchor row plus the next `headerRows - 1` rows. Column extent = every column whose merged header text is non-empty. Merged header text per column = non-empty normalized header-row cells joined top-down with a single space. Data rows run from below the header until the first run of **2 consecutive rows blank across the matched columns**, a row that anchors a later table, or sheet end.

**Header → columns.** Each `ColumnPlanSchema.from` must equal a merged header text (normalized-exact). Unmatched plan columns → fit/exec failure (`missing column:`). Raw headers matching no plan column and no `unpivot.valuePattern` are ignored and reported (`unknown column:`).

**Row filtering**, in order: fully-blank rows dropped; rows whose matched cells equal the header row (repeated page headers) dropped; `exclude` rules — a rule drops the row when the case-insensitive regex matches the raw text of its `column` cell (or any matched cell when `column: null`). Dropped rows are counted per rule with up to 3 sample rows in the report.

**Coercion** (`parse`): `number` — strip spaces and `,` thousands separators, parseFloat; `currency` — additionally strip a leading/trailing currency symbol (`$ € £`) ; `percent` — number then ÷ 100; `date` — Date instances (XLSX) → ISO; strings accepted as `YYYY-MM-DD`, `YYYY/MM/DD`, `M/D/YYYY` (US order), `D Mon YYYY`, `Mon D, YYYY` → `YYYY-MM-DD` TEXT. A cell that fails its parse becomes NULL and is counted (with ≤3 samples) in the report — coercion failures never abort ingest. Empty/whitespace cells → NULL (uncounted).

**Unpivot.** For each kept data row: one output row per matched value column — `keep` columns copied, `keyColumn` = the value column's merged raw header text (original casing, trimmed; always TEXT), `valueColumn` = the cell coerced by `valueParse`/`valueType`. Rows where the value cell is empty are skipped (no NULL value rows).

**Period validation.** When `periodColumn` is set: derive each row's period from the ISO date per granularity (quarterly `YYYY-Q⌈m/3⌉`, monthly `YYYY-MM`, yearly `YYYY`); rows whose derived period ≠ `slotPeriod` (or whose date failed to parse) are counted as mismatches; `onPeriodMismatch: "exclude"` drops them, `"keep"` ingests them with the slot period (all rows are always stamped with the slot period — validate-only).

**Report** (`ExecReport`): per table — resolved anchor `{sheet, row}` or null, `rowsKept`, `rowsExcluded` per rule with samples, `coercionFailures` per column `{count, samples}`, `periodMismatches` `{count, samples}`, `unknownColumns: string[]`. The report is the preview's data source and is persisted on confirm.

`fitParsePlan(grids, plan) → { verdict: "fit" | "partial" | "no_fit"; detail: string[] }` — anchoring + header matching only (no row processing). Detail lines (byte-exact): `unanchored table: <name>` · `missing column: <table>.<from>` · `unknown column: <raw>` (info; does not degrade the verdict). `fit` = every table anchored and every plan column matched; `no_fit` = no table anchored; else `partial`.

## 5. LLM contract (`packages/engine/src/proposePlan.ts`)

`proposeParsePlan({ client, filename, gridSample, families?, existingPlan?, fitDetail?, execReport?, feedback? }) → ParsePlan | null` — one structured call (same client/model env as everything else), zod-validated + `validateParsePlan`; one retry on invalid output, then null (→ no plan; classify proposal proceeds plan-less as today).

**gridSample** (≤6,000 chars): per sheet — `## sheet: <name> (<rows>×<cols>)` then the first 30 rows × first 15 columns as `R<n>: cell | cell | …` with raw text truncated to 20 chars/cell — followed by the island detector's `scanSample` output as `## detector hints`. Row numbers aid reasoning; plans themselves contain no row indices.

**Amendment invariant** (system prompt, enforced by review not code): when `existingPlan` is present, logical table names and canonical column names MUST be preserved; the job is re-anchoring and re-mapping (`amount_usd` → existing canonical `amount`). New canonical columns may be added (normal drift path). Removing a table or renaming canonicals is prohibited.

**Self-check round.** After any proposal, the server executes it on the file. The result is **degenerate** when: any table anchored 0 kept rows, OR any anchor unresolved, OR any mapped column's coercion failure rate is 100% over ≥1 attempted cell. On degeneracy, call `proposeParsePlan` once more with `execReport` attached; the second result stands (degenerate or not) and renders in the preview. Exactly one automatic round; the human feedback loop sits above it.

## 6. Flows

**Upload** — unchanged (multipart, 100MB, `unfiled`).

**Classify** (`classify/route.ts`, extended): for tabular files, after `classifySource` picks/creates the family proposal:
- Family has a plan → `fitParsePlan`. `fit` → execute stored plan for preview; no LLM. `partial`/`no_fit` → `proposeParsePlan` with `existingPlan` + `fitDetail` (+ self-check).
- No plan (new or legacy family) → `proposeParsePlan` from scratch (+ self-check).
- Proposal JSON on the source row gains: `plan` (the proposed/stored plan), `planStatus: "stored" | "proposed" | "amended" | "none"`, `preview` `{ tables: { name, columns, rows(≤8) }[] }`, `report` (ExecReport), plus the existing tables/skippedFragments/drift enrichment computed **from the plan's output schema** (drift vs warehouse unchanged in mechanism). `proposeParsePlan` returning null → `planStatus: "none"`, today's island path, today's chip.

**Replan** — new `POST /api/projects/[id]/sources/[sourceId]/replan` `{ feedback: string }`: re-runs `proposeParsePlan` with the current proposal's plan + `feedback` (+ self-check), re-executes, updates the stored proposal JSON, returns the same shape as classify. Errors surface as `{ error: "replan failed: <message>" }` (500); the prior proposal is untouched.

**Confirm** (`fileSource`, plan path): when the confirmed proposal carries a plan, inside the existing mutex + single transaction: persist the plan to `source_families.parse_plan` (also for amendments), `applySchema` from the plan's output schemas (types from `ColumnPlanSchema.type` / unpivot types; `_period` on periodic as today), `deleteRows` over existing ∪ plan tables for the slot, `loadGrids` + `executeParsePlan`, insert kept rows in 10k batches, persist the report to new nullable column `sources.parse_report`. If ALL tables yield 0 kept rows → rollback with `ingest failed: plan produced no rows` (a single empty table among non-empty ones is a report fact, not an error). Scan/attach throws keep the `ingest failed: <cause>` contract. Plan-less confirm = today's path, byte-identical.

**Steady state**: filing a new period into a settled family = fit-check + stored-plan execution + preview + one click. Zero LLM calls, fully deterministic.

## 7. Preview panel (Builder-side source manager, "Manuscript" idiom)

On the confirm card, plan-bearing proposals render a **Parsing** section:
- Plain-language step lines generated deterministically from the plan (`renderPlanSteps(plan)` in `apps/web/lib/planSteps.ts`): `header: rows 2–3 merged` · `excluding rows where <column|any cell> matches /<pattern>/i` · `<from> → <name> (<parse|type>)` · `unpivot: <matched headers> → <keyColumn>/<valueColumn>` · `period check: <column> vs slot (<keep|exclude> mismatches)`.
- Per table: name, first ≤8 post-transform rows (mono grid), `kept <n> · excluded <n> · coercion failures <n> · period mismatches <n>`; excluded/failure samples struck-through/`text-ink/40`; a keep/exclude toggle for period mismatches (updates `onPeriodMismatch` in the proposal before confirm).
- Feedback textarea + "Revise plan" → replan endpoint → panel refreshes. Confirm files the source with the shown plan.

## 8. Data model & upgrade

New nullable columns: `source_families.parse_plan` TEXT, `sources.parse_report` TEXT. Breaking DDL, no migration: `rm -f data/runoff.db && rm -rf data/warehouses && pnpm seed` (established pattern; openDb guard untouched). Plan-less families behave exactly as v1.3b — the island path, its tests, and its byte-exact strings are unchanged.

**Seed**: new quarterly family `ar_aging` for Meridian Retail with a deliberately messy fixture `ar_aging_q2_2026.xlsx` (genFixtures.ts, re-packed workbook.xml-first as required): title row glued above a 1-row header, a `Grand Total` row, currency-formatted `Amount Due ($)` column, plus a second wide sheet (months as columns) unpivoted by the plan. Seed pins the **literal ParsePlan JSON** (serving as reference documentation) on the family and ingests through the plan path.

## 9. What doesn't change

Warehouse schema evolution (union + one-way widening), slot/replacement semantics, `deleteRows`, the ingest mutex, `runWarehouseSql`/`formatSqlResult`, catalog, baked queries, locators/asserts, copilot tools, Reader/runs. Plans produce ordinary `fam_*` tables; nothing downstream knows they exist.

## 10. Error handling

LLM invalid plan → one retry → null → plan-less flow. Degenerate execution → one automatic self-check round. Replan failure → `replan failed: <message>`, prior proposal intact. Plan-path ingest is atomic under the existing transaction + mutex; all-empty output → `ingest failed: plan produced no rows`. Coercion failures and period mismatches are per-cell/per-row data facts (NULL / flagged), never errors.

## 11. Byte-exact strings

New: `unanchored table: <name>` · `missing column: <table>.<from>` · `unknown column: <raw>` · `ingest failed: plan produced no rows` · `replan failed: <message>` · `EVAL PARSE FAIL: <detail>`. Unchanged: `ingest failed: <cause>` · `no tables detected in file` · all v1.3a/v1.3b strings.

## 12. Testing

- **Executor goldens** (engine, inline grids): glued title; Grand Total exclusion; repeated page headers; 2-row merged header; unpivot (incl. new-month column matching `valuePattern` without refit; empty value cells skipped); each coercion incl. `"$1,234.56"`, `"12%"`, `"007"` (TEXT unless `parse: "number"`), each date format, failure→NULL+count; period derivation per granularity + keep/exclude; anchoring with renamed sheet, moved island, `minMatch` partial signatures, plan-order row consumption; region termination (2 blank rows / next anchor / sheet end).
- **Fit-check units**: fit / partial / no_fit with byte-exact detail lines. **validateParsePlan** units per rule.
- **Web**: classify-with-plan (stubbed LLM) incl. stored-plan fit path (no LLM call asserted), self-check round trigger, replan loop, plan-path `fileSource` against a real warehouse (tables/types/period stamping/report persisted), `plan produced no rows` rollback, preview payload shape, `renderPlanSteps` snapshots.
- **Live**: `pnpm eval:parse` — temp project, upload the messy seed fixture, real `proposeParsePlan`; assert the executed plan keeps exactly the known data-row count (Grand Total excluded), maps the currency column to REAL with 0 coercion failures, and anchors both tables — `EVAL PARSE FAIL: <detail>` otherwise. Existing evals unchanged and must stay green.

## 13. Accepted limitations

Plan-path CSV loads the full grid in memory (XLSX already does; 100MB cap bounds both). Date parsing assumes US month-first for ambiguous `M/D/YYYY`. Anchor matching is exact on normalized text (no fuzzy header matching — a reworded header is a refit, by design). One plan per family (a family whose periods need structurally different plans should be split into families).
