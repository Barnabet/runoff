# Runoff v1.3a — Real-World Data Plane (Warehouse + Catalog + Copilot SQL)

**Date:** 2026-07-19
**Status:** Approved design, pending implementation plan
**Depends on:** v1.2b (projects, source families, classify-confirm upload) — shipped on main at 529a687.

## 1. Overview

v1.2b's ingestion assumes fixture-shaped files: one clean table per sheet, headers in row 1, small enough to parse in memory per run. Real uploads (large AR/ERP CSV extracts, dashboard-style multi-table XLSX) break those assumptions silently — merged tables, dropped columns, polluted sums, and a 40-row prompt cap that hides 99% of a large file.

v1.3a builds the data plane that fixes ingestion and gives the copilot real data access:

1. **Island detection** — deterministic multi-table parsing of XLSX sheets; junk text dropped visibly.
2. **Project warehouse** — each filed tabular source is ingested into a per-project SQLite database, one stable table set per family, periodic rows stamped with `_period`.
3. **Schema catalog** — a compact families → tables → columns/types/row-counts view so agents know where to look without reading data.
4. **Copilot ad-hoc SQL** — a read-only `run_sql` tool with row/byte caps.

The **run pipeline is untouched** in v1.3a: drafting pack, checks dialect, and worker source resolution stay byte-identical. v1.3b (separate spec) moves runs onto the warehouse: baked `:period`-parameterized queries stored on sections, scalar-SQL checks/locators, retirement of the 40-row pack.

Scale target: **tens of MB / ~1M rows per file.** Ingestion must be streaming (bounded memory); querying must be sub-second for typical aggregations.

## 2. Scope

**In scope (v1.3a):**
- Island detection for XLSX; streaming ingest for CSV and XLSX.
- Per-project warehouse SQLite files; ingest/refile/delete lifecycle wired into `fileSource`.
- Schema-drift handling across periods (additive columns, missing columns, type widening) surfaced at confirm.
- Confirm-chip UI extension: detected tables, skipped fragments, drift diff.
- `catalog()` in core/engine-consumable form; copilot context switched from pack summaries to catalog.
- Copilot `run_sql` tool; `query_sources` inspect mode backed by catalog + samples.
- Upload size cap. New seed families + fixtures; `pnpm eval:sql`.

**Out of scope (v1.3b or later):**
- Any change to `buildSourcePack`, `packForPrompt`, drafting prompts, `checks.ts`, `dialect.ts`, worker `resolveSources`, or run events.
- Baked/named queries on sections; SQL checks/locators.
- CSV preamble/multi-table detection (a CSV is always one table, header = first row — documented limitation).
- DuckDB, sandboxed code execution, cross-project queries.

## 3. Warehouse

**Location:** one SQLite file per project at `<RUNOFF_WAREHOUSE_DIR>/<projectId>.db`, where `RUNOFF_WAREHOUSE_DIR` defaults to `data/warehouses`. The root `dev`/`seed`/`eval*` scripts set it alongside `RUNOFF_DB`/`RUNOFF_FILES_DIR`. Created lazily on first ingest. (The app has no project-deletion feature; orphaned warehouse files are not v1.3a's concern.)

**Ownership:** the app DB (`runoff.db`) keeps all metadata (families, sources, periods, status). The warehouse holds only data rows and is **rebuildable**: the original uploaded file on disk remains the source of truth and is never modified or deleted by ingestion.

**Table naming:**
- A family whose file yields exactly one table: `fam_<key>`.
- A family whose file yields multiple tables: `fam_<key>__<table_slug>`, one warehouse table per detected table. `<table_slug>` is the slugged sheet name; multiple islands on one sheet get positional suffixes: first island (top-to-bottom, then left-to-right) keeps the sheet slug, subsequent islands get `<sheet_slug>_2`, `_3`, …
- Slugs: lowercase, `[a-z0-9_]` only (other chars → `_`, runs collapsed), prefixed with `t_` if starting with a digit, deduped within the family by `_2` suffixing.

**Columns:**
- Header cells are slugged the same way; empty header cells within a table become `column_<N>` (1-based position); duplicate names after slugging get `_2` suffixes.
- `_period` is reserved: a source column slugging to `_period` becomes `_period_2`. Periodic families get `_period TEXT NOT NULL` stamped on every row; constant families have no `_period` column.
- Types inferred per column over the full data pass: all-integer → `INTEGER`, all-numeric → `REAL`, otherwise `TEXT`. Empty cells ingest as `NULL` and don't affect inference.

**Row identity:** none beyond SQLite rowid — the warehouse is an analytical copy, not a system of record.

## 4. Parsing & island detection

**CSV (`.csv`):** parsed with papaparse in streaming mode (`step` callback, `header: false` — the first row is taken as headers by the same rule as XLSX blocks). One table per file, always. Memory stays bounded at 1M rows (rows are batched into the insert transaction, 10k per batch).

**XLSX (`.xlsx`/`.xls`):** read with ExcelJS's streaming `WorkbookReader` (never `workbook.xlsx.readFile`). Per sheet, island detection runs on the non-empty-cell grid:

1. Split rows into bands separated by ≥1 fully blank row.
2. Within each band, split columns into runs separated by ≥1 fully blank column.
3. Each band × run rectangle is a candidate block. Its first row is the header row.
4. A block qualifies as a **table** if it spans ≥2 columns and ≥2 rows (header + ≥1 data row). Anything smaller is a **skipped fragment**: not ingested, but counted and shown (first 80 chars of its text) in the confirm UI.

Formula cells contribute their cached `result` value; rich text is flattened; dates become ISO strings (reuse the existing `cellText`/`cellValue` coercions from `sourcePack.ts`).

**Non-tabular files (PDF, DOCX, TXT, MD, JSON, unknown):** never touch the warehouse. They keep today's text-extraction path end to end and appear in the catalog as documents (§6).

**Determinism:** detection is pure — same file bytes, same result. It runs once at classify time (to render the confirm chip) and once at confirm time (to ingest); both runs use the same function.

## 5. Ingestion lifecycle & schema drift

Ingestion happens inside the existing `fileSource` transaction in `apps/web/lib/sourceManager.ts` — validation first (unchanged v1.2b rules), then warehouse writes, then the app-DB status flip. Because the warehouse is a separate SQLite file, `fileSource` gets cross-database atomicity by `ATTACH`ing it to the app-DB connection for the duration of the call (`ATTACH <path> AS wh` … writes against `wh.<table>` … `DETACH wh` in a `finally`): SQLite journals attached databases together, so one `ROLLBACK` reverts both. Any ingest error rolls back everything: no half-filed slots, byte-exact error surfaced (§9).

- **Filing a periodic slot:** for each detected table → `DELETE FROM <table> WHERE _period = ?` then bulk `INSERT` with `_period` stamped. Refiling (replacing the occupant) therefore swaps that period's rows atomically; the replaced file's bytes stay on disk (v1.2b provenance rule).
- **Filing a constant family:** `DELETE FROM <table>` (all rows), then insert.
- **Deleting a filed source:** removes its rows (`WHERE _period = ?`, or all rows for constants). Warehouse tables are never dropped in v1.3a (the app has no family-deletion feature); a table with zero rows is legal (renders as a gap in the catalog).

**Schema drift** (incoming file vs. the family's existing warehouse tables), computed at classify time and shown in the confirm chip:

| Drift | Handling |
|---|---|
| New column | `ALTER TABLE ADD COLUMN` — older periods read `NULL`. |
| Missing column | Allowed — incoming rows get `NULL` there. |
| Type conflict (e.g. INTEGER column, incoming TEXT) | Column widens: INTEGER → REAL → TEXT (one-way). Existing values are preserved (SQLite type affinity makes this a metadata change). |
| New table (sheet/island absent from previous periods) | New warehouse table created; earlier periods simply have no rows in it. |
| Missing table | Allowed — that table has no rows for this period. |

All five cases are informational, never blocking — the user sees the diff and confirms. There is no "reject on drift" mode in v1.3a.

## 6. Schema catalog

`catalog(db, projectId)` (new module `apps/web/lib/catalog.ts`, mirrored for the worker in v1.3b) reads warehouse metadata + the app DB and returns, per family:

```ts
interface CatalogFamily {
  id: string; key: string; label: string;
  kind: "periodic" | "constant";
  granularity: "quarter" | "month" | "year" | null;
  queryable: boolean;                    // false for document families
  tables: CatalogTable[];                // empty when !queryable
  filedPeriods: string[];                // ascending; [] for constants
}
interface CatalogTable {
  name: string;                          // warehouse table name
  columns: { name: string; type: "INTEGER" | "REAL" | "TEXT" }[];
  rowCounts: Record<string, number>;     // period → count; {"": n} for constants
}
```

**Serialized form** (built by the engine for prompts; ~2 lines per table):

```
marketing_spend — "Marketing spend" (periodic, quarter; filed: 2026-Q1, 2026-Q2)
  fam_marketing_spend(campaign TEXT, channel TEXT, spend REAL, region TEXT) — 1,204 rows (2026-Q1: 598, 2026-Q2: 606)
brand_guidelines — "Brand guidelines" (constant) — document, not queryable
```

The copilot's `CopilotContext` gains `catalog: CatalogFamily[]`; the system prompt's source section is built from the serialized catalog instead of pack summaries. `bound` markers and the `Not bound to this blueprint:` trailer behave exactly as in v1.2b.

## 7. Copilot SQL

**New tool `run_sql`** — `{ sql: string }` (required, single statement):

- Executes against the project warehouse on a dedicated read-only connection: opened with `{ readonly: true }` **and** `PRAGMA query_only = ON`.
- better-sqlite3 rejects multi-statement strings at `prepare()` — that error surfaces like any SQL error.
- Result caps: max **200 rows**; if the serialized result exceeds **10,000 characters**, rows are cut at the limit. Either truncation appends the exact line: `… truncated at <N> of <M> rows`.
- Result format: header row of column names, then `|`-separated rows.
- Errors return the byte-exact tool string `Tool error: sql: <sqlite error message>` — the model is expected to read the catalog and self-correct.
- Empty result: `(0 rows)`.

**`query_sources` changes:** the family-tree mode (no args) now renders from the catalog (same tree shape as v1.2b, plus table/column lines for queryable families). Inspect modes (`{familyId}` / `{familyId, period}`) return the family's catalog entry plus, per table, a 10-row sample via `SELECT * FROM <table> [WHERE _period = ?] LIMIT 10`. For document families, inspect keeps the v1.2b pack-based content dump (`packForPrompt(..., 20)`), and `Tool error: no file for <key> at <period>` is unchanged.

`compute` remains expression-only over the default pack (v1.2b recorded deviation) until v1.3b replaces it.

## 8. Upload flow & UI

- **Size cap:** POST `/api/projects/[id]/sources` rejects any file over **100 MB** with HTTP 413 and body `{ "error": "file exceeds 100MB limit" }` before writing anything.
- **Classify:** `classifySource` (engine) is unchanged in its LLM contract; its content sample for tabular files is now built from detected tables (headers + first rows of each) instead of raw bytes. The classify route additionally runs detection + drift and stores the result in the existing `sources.proposal` JSON under a new `tables` field: `{ tables: [{ name, columns, rowCount }], skippedFragments: number, drift: string[] }` (drift entries are human-readable lines, e.g. `new column: refund_flag (REAL)`).
- **Confirm chip (SourceManager):** below the family/period proposal, renders the detected-table breakdown (`<table_slug> — 12 cols · 48,112 rows`), a `skipped: <n> text fragment(s)` line when `skippedFragments > 0`, and drift lines in the existing mono style. Confirm/refile/delete flows, error surfacing, and chip-scoped errors from v1.2b are unchanged.
- **Family cards** show per-table row counts from the catalog instead of nothing (small mono line under the filed-period list).

## 9. Errors (byte-exact)

| Condition | Where | String |
|---|---|---|
| Upload over cap | POST sources route, HTTP 413 | `file exceeds 100MB limit` |
| Tabular file with zero detected tables filed into a periodic/constant family | `fileSource`, HTTP 400 | `no tables detected in file` |
| Warehouse ingest failure (disk, malformed data) | `fileSource`, HTTP 500 | `ingest failed: <cause>` (transaction rolled back) |
| SQL error in `run_sql` | copilot tool result | `Tool error: sql: <sqlite error message>` |
| `run_sql` on a project with no warehouse yet | copilot tool result | `Tool error: sql: no data ingested yet` |
| Inspect miss (unchanged from v1.2b) | copilot tool result | `Tool error: no file for <key> at <period>` |

All v1.2b `fileSource` validation strings are unchanged.

## 10. Upgrade path

No app-DB DDL change, so the v1.2b openDb guard is untouched. But existing filed sources have no warehouse rows, so after pulling v1.3a: **delete the DB file and `data/warehouses/`, then `pnpm seed`** (consistent with the v1.2b precedent). Re-filing each source by hand would also work; the docs recommend reseeding.

## 11. Testing, seed, evals

**Unit tests:**
- Island detection: two tables + a note line on one sheet; side-by-side tables split by a blank column; junk-only sheet (0 tables); 1×N and N×1 fragments skipped; empty header cells → `column_N`; slug collisions.
- Ingestion: periodic refile swaps only that period's rows; constant refile replaces all; delete removes rows; all five drift cases; type widening preserves values; `_period` collision renames; rollback on induced ingest failure leaves app DB and warehouse unchanged.
- Catalog: shapes above, including document families and zero-row tables.
- `run_sql`: read-only enforcement (INSERT/UPDATE/DROP each rejected), row cap, char cap with byte-exact truncation line, multi-statement rejection, empty result, no-warehouse error.
- Upload cap: 413 + byte-exact body.

**Seed:** adds to Meridian Retail — `ar_transactions` (periodic, quarter): a deterministic generated CSV (~10k rows: invoice id, customer, amount, issued/paid dates, status) per period via a seeded PRNG in the seed script; `regional_summary` (periodic, quarter): a committed fixture XLSX with two islands + a note line on one sheet, exercising detection in the seed path itself.

**Evals:** `pnpm eval` / `eval:copilot` / `eval:classify` must stay green. New root script `eval:sql` → `scripts/evalSql.ts`: seeds a project, runs a copilot turn asking a quantitative question over `ar_transactions`, asserts ≥1 successful `run_sql` tool call and a non-error final message.
