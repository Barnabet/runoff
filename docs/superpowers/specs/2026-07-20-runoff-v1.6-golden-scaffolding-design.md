## 1. Goal

Turn bound golden exemplars into blueprint sections through the builder copilot. A bound golden holds three things a blueprint needs: structure (document sections + headings), tone (verbatim prose), and app-verified `:period` SQL (binding inventories). v1.6 makes that lift explicit: the copilot gains a read tool that serves a deterministic **scaffold digest** of a bound golden, and composes ordinary section EditOps from it â€” verified binding SQL lifted verbatim into section queries, complementary agent SQL dry-run validated exactly as today.

Rides along: the **cleanup batch** â€” a client-safe `@runoff/core/client` exports subpath, catalog-builder dedupe (3 copies â†’ 1), boundness-counting dedupe (2 copies â†’ 1).

Decided in brainstorm: scope = scaffolding + cleanup batch (CSV/XLSX exemplar unification and cross-period value search stay backlogged); trigger = copilot tool only (no golden-card button); query policy = verified lift + agent fill.

## 2. Scaffold digest (engine, deterministic, zero LLM)

New `packages/engine/src/scaffoldGolden.ts`.

### 2.1 Types

```ts
export interface ScaffoldQuery {
  name: string;                                // binding item id (already ^[a-z][a-z0-9_]*$)
  sql: string;                                 // the verified binding SQL, verbatim
  provenance: "verified" | "verified-mismatch";
}
export interface ScaffoldSection {
  key: string;                                 // doc section key (already ^[a-z][a-z0-9_]*$)
  heading: string;
  prose: string;                               // serialized section content, capped (2.3)
  queries: ScaffoldQuery[];
  warnings: string[];
}
export interface ScaffoldDigest {
  goldenId: string;
  label: string;
  period: string | null;
  boundness: string;                           // engine boundnessLine output
  sections: ScaffoldSection[];
}
```

### 2.2 `buildScaffoldDigest(g: ScaffoldGoldenInput): ScaffoldDigest`

`ScaffoldGoldenInput` is the engine-side structural input `{ id, label, period, document: RunDocument, inventory: BindingInventory }` (the engine cannot import the web `ResolvedGolden` type; the web caller passes the resolved fields after its null checks). No catalog parameter â€” lifting reads only the golden's own bindings.

Precondition (enforced by the caller, Â§4): `golden.document !== null` and `golden.bindings !== null`. One digest section per document section, in document order.

Query lifting, per binding item anchored in the section (`item.anchor.sectionKey === section.key`):

- `binding.status === "bound"` â†’ `ScaffoldQuery` with `provenance: "verified"`.
- `binding.status === "mismatch"` â†’ lifted with `provenance: "verified-mismatch"` (the SQL executed and located the value; the golden's prose is what went stale) **and** a warning: `"<raw>" mismatches current data (golden says <raw> Â· data <verifiedValue>)`.
- `binding.status === "error"` or `binding === null` â†’ not lifted; warning: `"<raw>" has no data backing (<reason>)`.

Name dedupe within a section: item ids are unique per inventory, so collisions cannot occur between items; if a lifted name is not unique after lowering (defensive), append `_2`, `_3`, â€¦.

Section-level warning appended last when `queries.length === 0`: `no verified queries in this section`.

### 2.3 Prose serialization

Paragraph blocks: span texts joined with a space. Table blocks: `[table <N> cols Ã— <M> rows: <col> | <col> | â€¦]` â€” header line only, no data rows (bind-time rendering shows rows; scaffold tone needs shape, not data). Blocks joined with `\n`. Cap per section: **1500 chars**, truncated with a trailing `â€¦` on overflow.

### 2.4 `renderScaffoldDigest(d: ScaffoldDigest): string`

Prompt text, format:

```
SCAFFOLD DIGEST â€” golden "<label>" (period <period ?? "none">, <boundness>)

## section: <key> â€” <heading>
prose:
<prose>
queries:
  <name>: <sql>            [verified]
  <name>: <sql>            [verified-mismatch]
warnings:
  - <warning>
```

`queries:`/`warnings:` lines omitted when empty. SQL is NOT truncated in the digest (the agent must lift it verbatim; the existing 120-char truncation in `renderGoldenForPrompt` stays as-is for the read-golden path).

## 3. Copilot tool

### 3.1 Tool + ctx

### 3.2 System-prompt guidance

Appended to the copilot system prompt (goldens block): when the user asks to scaffold from a golden example, call `get_golden_scaffold`, then propose sections with the existing ops â€” `add_section` for structure and guidance (distill the golden's tone from the prose; do not copy prose verbatim into guidance), `update_section_queries` lifting `[verified]`/`[verified-mismatch]` SQL **verbatim** (names as given) plus any complementary queries you judge the narration needs; relay digest warnings to the user; prefer one section per op so the user can steer between them.

No new op types. Complementary queries flow through the existing `update_section_queries` dry-run validation unchanged.

## 6. Error handling / degradation

The v1.5 degradation contract extends to scaffolding: `scaffoldDigest` never throws â€” corrupt bindings already degrade to `bindings: null` inside `resolveGolden`, which lands in the `has no bindings` inert string; a document section with zero binding items still scaffolds (prose + `no verified queries in this section` warning). ## 11. Accepted limitations

- Tone transfer is prompt-mediated (guidance distilled by the copilot), not verified â€” quality rides on the model.
- Lifted queries are scalars/covering queries from bindings; narration-grade row-level queries are agent-composed (dry-run validated, but not provenance-verified).
- One digest per tool call; a very large golden (60 items Ã— long prose) yields a large tool result â€” bounded by the 1500-char prose cap and item cap (â‰¤60), worst case â‰ˆ 20k chars. The copilot's general tool-result clamp is 10,100 chars (sized to run_sql); `get_golden_scaffold` results are clamped at **20,000** chars instead so a large digest is not clipped mid-section.
- The catalog equality gate pins the web prompt only; worker/eval consumers are behavior-equivalent by construction (same helper) but not diff-pinned.
