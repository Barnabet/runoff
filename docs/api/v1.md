# Runoff v1 API contract

The authoritative contract for the Runoff backend HTTP surface. It covers the
**entire** API — including endpoints whose Python implementation lands in a
later phase (R2/R3) — because the contract is frozen up front and the backend
fills it in over the R1–R4 program.

- **Authority.** The current TypeScript route handlers under
  `apps/web/app/api/**` are the source of truth for every shape, status code,
  and error string here. This document is written by reading each handler; the
  Python (`backend/`) implementation must match it byte-for-byte on the wire.
- **Scope.** 23 distinct route paths (one per `route.ts` file). Each subsection
  below documents one path and all of its methods.
- **Drift guard.** See the final section — a pytest asserts the FastAPI app's
  mounted R1 route set equals a hand-maintained manifest.

---

## 1. Contract rules

*(Verbatim from the spec's §3.1, `docs/superpowers/specs/2026-07-23-runoff-r1-api-contract-python-core-design.md`.)*

- Base path **`/api/v1`**. Paths are the existing Next.js route paths verbatim,
  just prefixed — e.g. `/api/blueprints/:id/copilot` →
  `/api/v1/blueprints/:id/copilot`. No renaming, no resource remodeling.
- Request/response JSON shapes are the current TS shapes **verbatim**: camelCase
  field names, same nesting, same nullability. The TS route source is the
  authority; the doc is written by reading each route handler.
- Errors: `{"error": "<message>"}` with the status codes the TS routes use
  today (400 invalid body, 404 missing resource, 409 where currently used, 500
  unexpected).
- SSE endpoints (`GET /runs/:id/events`, copilot streaming) are documented in
  R1 with their exact event framing (`data:` lines, event names, terminal
  behavior) even though their Python implementation is R2/R3.
- No auth, no pagination, no rate limiting — single-user local app (YAGNI).
- The doc carries one table per endpoint: method, path, request shape, response
  shape, status codes, and the phase (R1/R2/R3) in which the Python
  implementation lands.

### Phase tags

Each endpoint carries the phase in which its Python implementation lands:

- **R1** — pure SQLite CRUD (no LLM, no file storage, no SSE). Implemented.
- **R2** — run pipeline, sources/warehouse ingestion (file uploads), SSE.
  Implemented.
- **R3** — copilot + goldens LLM surface (streaming, unify/bind, exemplar
  upload).

The **route manifest** (`backend/tests/test_api_manifest.py`) is the machine
check for which method+path pairs are live in the Python backend today; every
endpoint tagged R1 or R2 below appears there (the R2 sources + run-events routes
were mounted in the R2 program — see §5).

### Common conventions

- All request bodies are JSON (`content-type: application/json`) unless the
  endpoint is explicitly documented as multipart.
- Body parsing failure returns **400 `{"error": "invalid JSON body"}`** on
  JSON endpoints, with one exception: `PATCH /goldens/:id`, whose TS handler does
  not guard the parse and 500s on malformed JSON (the Python backend returns the
  standard 400 — see §4.2).
- Multipart parsing is guarded only on `POST /projects/:id/sources`, which
  returns **400 `{"error": "expected multipart form data"}`** when the body is
  not a valid form. The multipart `POST /blueprints/:id/goldens` variant does not
  guard `formData()` in TS (a malformed form throws → framework 500), and in the
  Python backend that path is a 501 (§4.1) regardless.
- Path params are written `:name`. Query params are noted per endpoint.
- Field nullability: a type written `string | null` is always present but may be
  `null`; a field marked *optional* may be absent from the object entirely.
- Timestamps (`createdAt`, `finishedAt`, …) are SQLite datetime strings.

## 2. Endpoints

### 2.1 `GET, POST /projects`  — **R1**

#### `GET /api/v1/projects`

List every project with its blueprint count and most recent activity.

- **Request:** none.
- **Response 200:** `{ "projects": ProjectListItem[] }`

```
ProjectListItem {
  id: string
  name: string
  blueprintCount: number
  lastActivityAt: string | null   // latest run or revision timestamp
}
```

#### `POST /api/v1/projects`

Create a project. `name` is trimmed and required.

- **Request:** `{ name: string }`
- **Response 200:** `{ "id": string }`
- **Errors:**
  - 400 `{"error": "invalid JSON body"}`
  - 400 `{"error": "name required"}` — missing/empty/non-string `name` after trim.

---

### 2.2 `GET, PATCH /projects/:id`  — **R1**

#### `GET /api/v1/projects/:id`

The project header plus its scoped blueprint ledger and source-manager state.

- **Request:** none.
- **Response 200:**

```
{
  project: { id: string, name: string, createdAt: string },
  blueprints: BlueprintListItem[],   // see §2.8
  families: FamilySummary[],         // see §2.3
  unfiled: ProjectSourceRow[],       // see §2.3
  memories: MemoryRow[]              // project-scoped rows; see §2.12
}
```

- **Errors:** 404 `{"error": "project not found"}`

> Source note: `apps/web/lib/api.ts`'s `getProject` type omits the `memories`
> field, but the handler (`getProjectPayload`) returns it. The handler wins.

#### `PATCH /api/v1/projects/:id`

Rename the project.

- **Request:** `{ name: string }` (trimmed, required)
- **Response 200:** `{ "ok": true }`
- **Errors** (order as handled):
  - 404 `{"error": "project not found"}` — checked **before** body parse.
  - 400 `{"error": "invalid JSON body"}`
  - 400 `{"error": "name required"}`

### 2.3 `GET /projects/:id/sources` (R1) · `POST` multipart (R2, implemented)

#### `GET /api/v1/projects/:id/sources`  — **R1**

The project's source families plus unfiled uploads.

- **Request:** none.
- **Response 200:** `{ families: FamilySummary[], unfiled: ProjectSourceRow[] }`
- **Errors:** 404 `{"error": "project not found"}`

```
FamilySummary {
  id: string
  key: string
  label: string
  kind: "periodic" | "constant"
  granularity: ("month" | "quarter" | "year") | null
  filedPeriods: string[]
  filedEntries: { period: string, sourceId: string, name: string }[]
  liveFile: { sourceId: string, name: string } | null
  tables: { name: string, rowCount: number }[]
}

ProjectSourceRow {
  id: string
  projectId: string
  familyId: string | null
  period: string | null
  name: string
  kind: string
  storedFilename: string
  mime: string
  size: number
  status: "unfiled" | "filed" | "replaced"
  proposal: ClassifyProposal | null
  uploadedAt: string
  filedAt: string | null
}
```

#### `POST /api/v1/projects/:id/sources`  — **R2** (multipart upload) · implemented

Upload one or more files under the `files` key; each becomes an `unfiled`
source row and its bytes are stored under `RUNOFF_FILES_DIR`.

- **Request:** `multipart/form-data`, repeated `files` parts. Max 100 MB per file.
- **Response 200:** `{ sources: { id, name, storedFilename, mime, size, status }[] }`
  (`status` is always `"unfiled"`).
- **Errors:**
  - 404 `{"error": "project not found"}` — checked first.
  - 400 `{"error": "expected multipart form data"}`
  - 400 `{"error": "files are required"}` — no `File` parts present.
  - 413 `{"error": "file exceeds 100MB limit"}` — any file over the limit
    (validated before any byte is written).

### 2.4 `PATCH, DELETE /projects/:id/sources/:sourceId`  — **R2** · implemented

#### `PATCH /api/v1/projects/:id/sources/:sourceId`

Refile a source into a (possibly new) family slot. Same body + rules as
`confirm` (§2.6), delegated to `fileSource`.

- **Request:**

```
{
  familyId?: string,
  newFamily?: { key: string, label: string,
                kind: "periodic" | "constant",
                granularity: Granularity | null },
  period: string | null
}
```

- **Response 200:** `{ "ok": true }`
- **Errors:**
  - 400 `{"error": "invalid JSON body"}`
  - Any `{status, error}` returned by `fileSource` — status is the value that
    helper chose (400/404/409), `error` its message string (slot-rule messages
    live in the sources service, R2).

#### `DELETE /api/v1/projects/:id/sources/:sourceId`

Remove an unfiled or filed source and its stored file; `replaced` rows are kept.

- **Request:** none.
- **Response 200:** `{ "ok": true }`
- **Errors:**
  - 404 `{"error": "source not found"}`
  - 400 `{"error": "replaced sources cannot be deleted"}`

### 2.5 `POST /projects/:id/sources/classify`  — **R2** (LLM) · implemented

For each requested `sourceId` that is still an `unfiled` row in this project,
sample its content and ask the engine where it belongs; persist the proposal (or
leave NULL). Only the ids passed in `sourceIds` are considered — other unfiled
rows are untouched. Rows that fail to parse or classify are simply left without a
proposal, never erroring the batch.

- **Request:** `{ sourceIds: string[] }`
- **Response 200:** `{ sources: { id: string, proposal: ClassifyProposal | null }[] }`
- **Errors:**
  - 404 `{"error": "project not found"}` — checked first.
  - 400 `{"error": "invalid JSON body"}`

---

### 2.6 `POST /projects/:id/sources/confirm`  — **R2** · implemented

File an unfiled source into a family slot (creating the family when `newFamily`
is given). Delegates slot rules to `fileSource`.

- **Request:**

```
{
  sourceId: string,             // required
  familyId?: string,
  newFamily?: { key, label, kind, granularity },
  period: string | null,
  periodMismatch?: "keep" | "exclude"
}
```

- **Response 200:** `{ "ok": true }`
- **Errors:**
  - 400 `{"error": "invalid JSON body"}`
  - 400 `{"error": "sourceId is required"}` — non-string `sourceId`.
  - 400 `{"error": "periodMismatch must be \"keep\" or \"exclude\""}` — any other value.
  - Any `{status, error}` from `fileSource` (R2).

### 2.7 `POST /projects/:id/sources/:sourceId/replan`  — **R2** (LLM) · implemented

Revise the current plan proposal with user feedback; the prior proposal is only
replaced on success.

- **Request:** `{ feedback: string }` (non-empty after trim)
- **Response 200:** `{ proposal: ClassifyProposal }`
- **Errors:**
  - 400 `{"error": "invalid JSON body"}`
  - 400 `{"error": "feedback is required"}` — missing/empty `feedback`.
  - 404 `{"error": "source not found"}` — no matching `unfiled` row in this project.
  - 400 `{"error": "source has no plan proposal"}` — no stored proposal or the source is non-tabular.
  - 500 `{"error": "replan failed: no plan produced"}` — planner returned no plan.
  - 500 `{"error": "replan failed: <message>"}` — any thrown error during replan.

### 2.8 `GET, POST /blueprints`  — **R1**

#### `GET /api/v1/blueprints?projectId=…`

Every blueprint in one project with its bound-source count and latest run.

- **Request:** query param `projectId` (required).
- **Response 200:** `{ "blueprints": BlueprintListItem[] }`

```
BlueprintListItem {
  id: string
  name: string
  clientName: string
  cadenceLabel: string
  status: string
  currentRev: number
  sourceCount: number
  lastRun: {
    id: string
    finishedAt: string | null
    status: string
    openFlags: number
  } | null
}
```

- **Errors:** 400 `{"error": "projectId is required"}`

#### `POST /api/v1/blueprints`

Create a blueprint (scoped to a project) plus revision 1 with a default,
schema-valid `BlueprintContent`.

- **Request:** `{ name: string, clientName?: string, projectId: string }`
  (`name` trimmed+required; `clientName` defaults to `""`).
- **Response 200:** `{ "id": string }`
- **Errors:**
  - 400 `{"error": "invalid JSON body"}`
  - 400 `{"error": "name is required"}`
  - 400 `{"error": "projectId is required"}`
  - 400 `{"error": "unknown projectId"}` — no such project row.

### 2.9 `GET, PATCH /blueprints/:id`  — **R1**

#### `GET /api/v1/blueprints/:id`

The blueprint row, its current revision content (parsed), the owning project's
families, the bound family ids, the project, and per-section query row counts.

- **Request:** none.
- **Response 200:**

```
{
  blueprint: {
    id: string, name: string, clientName: string,
    cadenceLabel: string, status: string, currentRev: number,
    createdAt: string, projectId: string
  },
  content: BlueprintContent | null,   // null if the current revision row is missing
  project: { id: string, name: string },   // { id, name: "" } if the project row is gone
  families: FamilySummary[],
  boundFamilyIds: string[],
  queryRowCounts: Record<string, Record<string, number | null>>   // {} when content is null
}
```

- **Errors:** 404 `{"error": "blueprint not found"}`

> Source note: `apps/web/lib/api.ts`'s `getBlueprint` type omits
> `queryRowCounts` and the `blueprint.projectId` field; both are present in the
> handler response. The handler wins.

#### `PATCH /api/v1/blueprints/:id`

Update any of `name` / `clientName` / `cadenceLabel` / `status`, and when
`familyIds` is present, replace the `blueprint_families` rows. Family-set
validation runs before the write.

- **Request:** `Partial<{ name, clientName, cadenceLabel, status: string, familyIds: string[] }>`
- **Response 200:** `{ "ok": true }`
- **Errors:**
  - 404 `{"error": "blueprint not found"}` — checked before body parse.
  - 400 `{"error": "invalid JSON body"}`
  - 400 `{"error": "unknown family for this project"}` — a `familyIds` entry
    is unknown or belongs to another project.
  - 400 `{"error": "granularity differs among bound periodic families"}` — the
    bound periodic families don't share one granularity.

### 2.13 `GET /blueprints/:id/copilot` (R1) · `POST` (R3)

#### `GET /api/v1/blueprints/:id/copilot`  — **R1**

The blueprint's copilot conversation (both roles), ordered `rowid`.

- **Request:** none.
- **Response 200:** `{ "messages": CopilotMessage[] }`

```
CopilotMessage {
  id: string
  role: "user" | "assistant"
  body: string
  actions: CopilotAction[]     // [] when the row's actions column is NULL
  status: "ok" | "failed"
  createdAt: string
}
```

`CopilotAction` is one of:
`{ kind: "tool", tool: string, label: string }`,
`{ kind: "edit", op: EditOp }`, or
`{ kind: "memory", memoryId: string, body: string }`.

#### `POST /api/v1/blueprints/:id/copilot`  — **R3** (LLM, streaming SSE)

Run one copilot turn against the draft and stream the result as SSE. See the
SSE appendix §3.2 for the full event framing.

- **Request:** `{ message: string, draft: BlueprintContent, selectedKey?: string | null }`
- **Response 200:** `text/event-stream` (see §3.2). The assistant message is
  persisted; a mid-stream failure persists a `status: "failed"` row and emits an
  `error` event.
- **Errors** (JSON, before the stream opens):
  - 400 `{"error": "invalid JSON body"}`
  - 400 `{"error": "message is required"}` — missing/empty `message`.
  - 400 `{"error": "invalid draft"}` — `draft` fails `BlueprintContentSchema`.

### 2.14 `GET` + `POST` `/blueprints/:id/goldens`

#### `GET /api/v1/blueprints/:id/goldens`  — **R1**

List the blueprint's goldens.

- **Request:** none.
- **Response 200:** `{ "goldens": GoldenRow[] }`

```
GoldenRow {
  id: string
  blueprintId: string
  kind: "run" | "section" | "exemplar"
  runId: string | null
  sectionKey: string | null
  name: string | null
  mime: string | null
  storedFilename: string | null
  note: string | null
  period: string | null
  document: string | null      // RunDocument JSON (exemplars)
  unifyError: string | null
  bindings: string | null      // BindingInventory JSON
  createdAt: string
}
```

#### `POST /api/v1/blueprints/:id/goldens` — JSON variant  — **R1**

Star a run or section as a golden. Splits by content type: this is the JSON
(non-multipart) path. It inserts the golden, copies the run's period, and builds
the deterministic inventory via `rebuildRunGoldenInventory` (no LLM).

- **Request:** `{ kind: "run" | "section", runId: string, sectionKey?: string, note?: string }`
- **Response 200:** `{ "id": string }`
- **Errors:**
  - 400 `{"error": "invalid JSON body"}`
  - 400 `{"error": "kind ('run'|'section') and runId are required"}`
  - 400 `{"error": "sectionKey is required for kind 'section'"}`
  - 404 `{"error": "run not found for this blueprint"}` — no matching run, or the
    run belongs to another blueprint.

#### `POST /api/v1/blueprints/:id/goldens` — multipart variant  — **R3**

When `content-type` is `multipart/form-data`, the TS route stores an uploaded
exemplar file (`file`, optional `name`, `note`) and triggers the LLM
unify+bind pipeline.

- **Request:** `multipart/form-data` with a `file` part (+ optional `name`, `note`).
- **Response 200:** `{ "id": string }`
- **Errors (TS):** 400 `{"error": "file is required"}` — no `file` part.
- **R1 divergence:** the Python backend does not implement exemplar upload until
  R3 and returns **501** for the multipart variant. See §4.

### 2.15 `POST /blueprints/:id/goldens/:goldenId/unify`  — **R3** (LLM)

Re-run unify on an exemplar golden.

- **Request:** none (body ignored).
- **Response 200:** `{ "golden": GoldenRow }`
- **Errors:**
  - 404 `{"error": "golden not found"}` — no such golden, or not on this blueprint.
  - 400 `{"error": "only exemplar goldens can be unified"}`.

### 2.16 `POST /blueprints/:id/goldens/:goldenId/bind`  — **R3** (LLM)

Bind an exemplar golden (or rebuild a run/section golden's inventory). Optional
`feedback` steers an exemplar bind. An already-bound exemplar called with no
`feedback` runs `verifyStoredInventory` only — no LLM re-bind; the wire shape is
identical and the stored bindings are re-stamped, not re-derived.

- **Request:** `{ feedback?: string }` (body may be `{}` or empty; parse
  failure is swallowed → no feedback).
- **Response 200:** `{ "golden": GoldenRow }`
- **Errors:**
  - 404 `{"error": "golden not found"}` — no such golden, or not on this blueprint.
  - 400 `{"error": "feedback requires an exemplar golden"}` — `feedback` given
    for a run/section golden.
  - 400 `{"error": "golden is not unified"}` — exemplar has no `document` yet.
  - 500 `{"error": "<bind error>"}` — `bindExemplar` failed.

### 2.17 `PATCH, DELETE /goldens/:id`  — **R1**

#### `PATCH /api/v1/goldens/:id`

Set the golden's period (nullable). A missing `period` key, a non-object body,
or a non-matching value all 400. Re-verifies/rebuilds the inventory afterward.

- **Request:** `{ period: string | null }`
- **Response 200:** `{ "golden": GoldenRow }`
- **Errors:**
  - 400 `{"error": "invalid JSON body"}`
  - 400 `{"error": "invalid period: undefined"}` — non-object body or missing `period` key.
  - 400 `{"error": "invalid period: <value>"}` — a non-null value that doesn't
    match any period pattern.
  - 404 `{"error": "golden not found"}`

  > Divergence: the TS handler does **not** wrap `req.json()` in a try/catch, so
  > malformed JSON throws and yields a framework **500** rather than a 400. The
  > Python backend returns 400 `{"error": "invalid JSON body"}` here (a
  > deliberate, benign hardening). See §4.2.

#### `DELETE /api/v1/goldens/:id`

Delete the golden and best-effort unlink its stored file.

- **Request:** none.
- **Response 200:** `{ "ok": true }`
- **Errors:** 404 `{"error": "golden not found"}`

### 2.22 `GET /runs/:id/events`  — **R2** (SSE) · implemented

Server-Sent Events: replays the run's existing event log, then tails new rows
until a terminal event. Full framing in the SSE appendix §3.1.

- **Request:** none.
- **Response 200:** `text/event-stream` (see §3.1).
- **Errors:** none — an unknown run id yields an empty stream that stays open
  (heartbeats only) until the client disconnects; the loop never breaks without a
  terminal event.

### 2.23 `POST /runs/:id/inputs`  — **R1**

Queue an out-of-band input for a running run (`pause`/`resume`/`steer`/`answer`).
Re-answering an unconsumed question **replaces** the pending row rather than
inserting a second one.

- **Request:** `{ kind: "pause" | "resume" | "steer" | "answer", text?: string, questionId?: string }`
- **Response 200:** `{ "ok": true }`
- **Errors:**
  - 400 `{"error": "invalid JSON body"}`
  - 400 `{"error": "kind must be one of pause|resume|steer|answer"}`
  - 404 `{"error": "run not found"}`

## 3. SSE appendix

Both streaming endpoints respond with the same headers:

```
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive
```

Every SSE message is a single `data:` line followed by a blank line
(`data: <payload>\n\n`). Heartbeats are SSE comment lines (`: ...\n\n`) and
carry no data. There are no named `event:` types — the discriminator lives
inside each JSON payload's `type` field.

### 3.1 `GET /runs/:id/events`  (R2, implemented)

Source: `apps/web/app/api/runs/[id]/events/route.ts`.

**Framing.** Each event row is emitted as its raw stored JSON payload:

```
data: <runEvents.payload JSON>\n\n
```

The payload is a `RunEvent` object (its `type` field is the discriminator).
`RunEvent` variants:

```
{ type: "run_started", sectionKeys: string[], blueprintRev: number,
  memoryIds?: string[], period?: string, gaps?: string[] }
{ type: "source_read", sourceId: string, label: string, summary: string }
{ type: "section_started", sectionKey: string }
{ type: "text_delta", sectionKey: string, text: string }
{ type: "section_completed", sectionKey: string, blocks: Block[],
  words: number, ms: number, retries: number }
{ type: "section_failed", sectionKey: string, error: string }
{ type: "check_passed", sectionKey: string, rule: string }
{ type: "check_failed", sectionKey: string, rule: string, detail: string }
{ type: "retry_started", sectionKey: string, reason: string }
{ type: "question_raised", questionId: string, sectionKey: string,
  question: string, options: string[], fallback: string, deadlineSection: string }
{ type: "question_answered", questionId: string, answer: string }
{ type: "question_fallback_applied", questionId: string }
{ type: "flag_raised", flagId: string, code: string, sectionKey: string,
  question: string, options: string[] }
{ type: "steer_received", text: string }
{ type: "paused" }
{ type: "resumed" }
{ type: "render_started" }
{ type: "run_completed", stats: RunStats, document: RunDocument }
{ type: "run_failed", error: string }
{ type: "log", level: "info" | "warn" | "error" | "user", message: string }
```

**Backlog replay.** On connect, `last = 0`; the loop selects all `run_events`
rows for the run with `seq > last`, ordered by `seq`, and emits each — so an
already-finished run replays its entire log immediately. `last` advances to the
highest `seq` sent.

**Tail loop.** After the backlog it polls every **200 ms** for rows with
`seq > last`. A loop counter is incremented every iteration regardless of whether
rows were sent, and on every 75th iteration (≈ every 15 s) it emits a heartbeat
comment `: ping\n\n`.

**Terminal close.** When a batch contains a `run_completed` or `run_failed`
event, the loop breaks after emitting that batch and the stream closes. It also
stops when the client aborts (`req.signal` abort). There is no explicit "close"
data frame — the connection simply ends.

### 3.2 `POST /blueprints/:id/copilot`  (R3)

Source: `apps/web/app/api/blueprints/[id]/copilot/route.ts`.

**Framing.** Each engine event is JSON-encoded on one `data:` line:

```
data: <CopilotEvent JSON>\n\n
```

`CopilotEvent` variants (the `type` field is the discriminator):

```
{ type: "text_delta", text: string }        // incremental assistant text
{ type: "tool_activity", label: string }     // tool-use progress label
{ type: "edit", op: EditOp }                  // an applied Builder edit op
{ type: "memory_saved", memoryId: string, body: string }
{ type: "done", messageId: string }           // terminal — success
{ type: "error", message: string }            // terminal — failure
```

**Terminal events.** Exactly one of `done` or `error` ends the stream, after
which the server calls `close()`:

- On success the assistant reply + actions are persisted, then
  `{ type: "done", messageId }` is emitted (`messageId` is the new
  `copilot_messages` row id).
- On failure the partial streamed text + actions are persisted as a
  `status: "failed"` assistant row, then `{ type: "error", message }` is emitted.

There are no heartbeat comments on this stream; it runs to a terminal event.

## 4. Divergences

Two intentional wire-contract divergences exist between this contract and the
Python backend (§4.1, §4.2) — both documented here and enforced by the backend
code — plus one sanctioned near-parity note on PDF text extraction (§4.3) that
affects prompt text only, never a wire shape.

### 4.1 Multipart `POST /blueprints/:id/goldens` → 501 until R3

The TS route (§2.14) accepts a multipart exemplar upload and runs the LLM
unify+bind pipeline. That pipeline is R3. Until then the Python backend detects
`multipart/form-data` and declines:

- **Status:** `501`
- **Body:** `{"error": "exemplar upload not yet implemented in this backend (R3)"}`

Source: `backend/runoff_api/api/goldens.py`. The JSON variant of the same path
is fully implemented in R1 (§2.14).

### 4.2 `PATCH /goldens/:id` malformed JSON → 400 (TS: framework 500)

The TS handler (`apps/web/app/api/goldens/[id]/route.ts:11`) reads the body as
`const { period } = (await req.json()) as …` with **no** try/catch, so a
malformed JSON body throws and Next.js returns a framework **500**. The Python
backend wraps the parse and returns the standard **400
`{"error": "invalid JSON body"}`** instead. This is a deliberate, benign
divergence (a hardening that only affects the malformed-body case; every
well-formed request matches TS byte-for-byte). Source:
`backend/runoff_api/api/goldens.py`.

### 4.3 PDF text extraction is near-parity (pypdf vs pdf-parse)

Tabular ingestion is byte-exact across stacks — the CSV (papaparse → Python
`csv`) and XLSX (ExcelJS → openpyxl) ports are pinned cell-for-cell, so every
warehouse table, row count, and figure matches TS exactly (this is what the diff
and parity harnesses verify). PDF sources are the one sanctioned exception: the
Python backend extracts a PDF's text with **pypdf**
(`backend/runoff_api/engine/source_pack.py`), whereas the TS stack uses
**pdf-parse**. The two libraries emit slightly different text for the same PDF
(whitespace, line breaks, glyph/ligature handling), so the extracted body a PDF
contributes to a drafting prompt — and any drafted prose or run-time summary
derived from it — can differ between stacks. The scope is narrow and deliberate:

- **PDF sources only.** DOCX uses the same `mammoth` library on both stacks;
  csv/xlsx never reach the prompt (the warehouse owns them).
- **Prompt text only, never a wire shape.** The `source_read` event's summary
  line for a PDF is size-derived (`"<name> — PDF, <kb> KB (read at run time)"`)
  and stays identical; it is the extracted `text` feeding the LLM that diverges.
  No JSON field's shape, key, or type differs — only the free-text content a
  model then reasons over.

This is fuzzy document text that feeds prompts, never byte-exact warehouse data,
so it is out of scope for the byte-for-byte diff/parity guarantee (spec §6).

Apart from the §4.1/§4.2 wire-contract cases and the §4.3 PDF-text caveat, every
R1- and R2-tagged endpoint matches the TS wire contract exactly and is mounted in
the Python app; only R3-tagged endpoints remain unmounted (they return the
framework's default 404 until their phase lands).

## 5. Drift guard

The set of method+path pairs the FastAPI app actually mounts is asserted equal
to a hand-maintained manifest by
[`backend/tests/test_api_manifest.py`](../../backend/tests/test_api_manifest.py):

- `R1_READ_ROUTES` ∪ `R1_WRITE_ROUTES` = `R1_ROUTES` — the 23 R1 method+path
  pairs (10 GET reads + 13 writes).
- `R2_SOURCES_ROUTES` (6: the multipart upload, refile, delete, classify,
  confirm, and replan writes) ∪ `R2_EVENTS_ROUTES` (1: the run-events SSE GET) —
  the 7 R2 pairs mounted in the R2 program.
- `REQUIRED_ROUTES` = `R1_ROUTES` ∪ `R2_SOURCES_ROUTES` ∪ `R2_EVENTS_ROUTES` —
  the 30 method+path pairs the Python backend serves today.
- `test_route_manifest()` enumerates every mounted `APIRoute` and asserts it
  equals `REQUIRED_ROUTES`. Adding, removing, or renaming a route without
  updating the manifest fails the test.

Every endpoint tagged **R1** or **R2** in this document appears in that manifest;
the manifest is the machine-checkable subset of this contract. FastAPI's
generated `/openapi.json` is a debugging aid only — **this document is
authoritative.**
